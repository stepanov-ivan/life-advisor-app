## Context

Текущее логирование питания построено вокруг slot-based UX в `DashboardView`: пользователь выбирает конкретный слот, вводит текст для этого слота, а затем при необходимости открывает отдельный editor для ручной правки. Этот поток хорошо работает для прямого ввода в один приём пищи, но плохо масштабируется на agent-first сценарии, где пользователь хочет свободно описать еду за день, за несколько дней или попросить внести правку без выбора конкретной карточки заранее.

Из proposal зафиксированы следующие продуктовые ограничения:

- новый вход в логирование должен начинаться из одной глобальной CTA-кнопки агента, доступной поверх основного приложения;
- первый этап поддерживает только сценарии `meal-create` и `meal-edit`;
- сообщение сначала проходит `scenario classification`, затем `plan building`, затем переводится в execution mode;
- агент подготавливает meal draft, но пользователь подтверждает каждый шаг отдельно;
- реальные `MealEvent` создаются или обновляются только после подтверждения конкретного шага;
- неподтверждённый хвост цепочки не сохраняется и не восстанавливается;
- после планирования `date` и `mealSlot` не редактируются напрямую пользователем.

Технический контекст тоже важен: в приложении уже есть модели `MealEvent`, `MealWindow`, `EstimateItem`, текущий pipeline оценки через `LLMClient`, а также логика пересчёта правил и day-based dashboard. Поэтому change должен не ломать существующую доменную модель питания, а заменить только способ оркестрации пользовательского ввода до момента сохранения `MealEvent`.

## Goals / Non-Goals

**Goals:**

- Перевести UX логирования питания с slot-based ввода на agent-first вход через один глобальный чатовый composer.
- Отделить распознавание сценария от построения шагов и от выполнения шага в UI.
- Заложить общий agent foundation, который в будущем сможет обслуживать не только meal domain, но и правила, аналитику и настройки.
- Ввести execution mode на дашборде без создания отдельного экрана, чтобы пользователь видел карточки дня и мог подтверждать шаги в контексте.
- Сохранять текущую доменную модель `MealEvent` и существующий pipeline оценки/пересчёта после подтверждения шага.
- Поддержать multi-step и multi-day планы, но с детерминированным порядком выполнения и явной группировкой по дням.
- Ограничить локальные правки внутри шага детерминированными действиями над продуктами и порциями.

**Non-Goals:**

- Полностью автономное применение действий без подтверждения пользователя.
- Сохранение и восстановление незавершённых execution chains между сессиями.
- Поддержка смешанных create/edit цепочек в V1.
- Прямое ручное редактирование `date` и `mealSlot` после построения шага.
- Переписывание существующей persisted модели питания или rule engine под новый формат данных.

## Decisions

### 1. Архитектура делится на три слоя: classifier, planner, execution runtime

**Decision:** Ввести общий agent orchestration слой, который делит процесс на:

- `AgentScenarioRouter`: сначала выбирает домен агента (`meal`, в будущем также `rules`, `analytics`, `settings`);
- доменный classifier, который распознаёт сценарий внутри выбранного домена;
- доменный planner, который превращает сообщение в цепочку нормализованных шагов;
- общий execution shell/runtime, который управляет execution mode, активным шагом, resolve-состояниями и commit-ами;
- доменные step renderers, которые рисуют содержимое текущего шага внутри общего shell.

Для V1 router фактически ведёт только в meal domain, но структура не должна проектироваться как одноразовый meal-only runtime.

Для meal domain первый classifier/planner слой выглядит так:

- `MealScenarioClassifier`: распознаёт `meal-create` или `meal-edit`, а также грубый временной контекст;
- `MealStepPlanner`: превращает сообщение в цепочку нормализованных шагов;
- `MealExecutionRuntime`: доменная часть runtime для meal draft-ов и meal-level commit-ов.

Шаги имеют только два базовых типа:

- `create(date, mealSlot, productsDraft)`
- `edit(date, mealSlot, productsDraft)`

`resolve` не является отдельным типом шага. Это transient state шага, который возникает при конфликте, неоднозначном target-е или нехватке обязательных аргументов.

**Why:** Мы уже договорились, что “классификатор делает всё” будет слишком хрупкой границей. Разделение на router/classifier/planner/runtime упрощает дальнейшее добавление новых сценариев и доменов, не смешивая UX, LLM parsing и локальный state management.

**Alternatives considered:**

- Один LLM-вызов, который сразу возвращает финальные операции для сохранения. Отклонено: плохо укладывается в confirm-first UX и не даёт чистой модели execution mode.
- Классификатор, который сам строит всю цепочку. Отклонено: responsibility становится слишком широкой.
- Meal-only orchestration без общего router/shell фундамента. Отклонено: плохо масштабируется на следующие агентные сценарии.

### 2. Execution mode живёт поверх дашборда, а не на отдельном экране

**Decision:** Execution mode реализуется как общий agent execution shell, но для каждого домена у него есть свой home surface. Для meal domain home surface — `DashboardView`.

После успешного планирования meal-сценария приложение не открывает новый flow screen. Вместо этого shell:

- при необходимости переключает пользователя на tab дашборда;
- переводит `DashboardView` в специальный execution mode;

- чатовый composer скрывается;
- внизу показывается progress bar шагов с группировкой по дням;
- появляется кнопка закрытия mode с confirm-алертом, если есть неподтверждённые шаги;
- активный шаг визуально фокусирует соответствующую карточку приёма пищи текущего дня.

Если цепочка multi-day, runtime автоматически переключает `selectedDate` при переходе к первому шагу следующего дня.

В каждый момент времени в приложении может существовать только одна активная agent session:

- один открытый chat/composer;
- одна execution chain;
- один активный домен;
- один общий progress shell.

**Why:** Пользователь должен работать в контексте карточек питания, а не покидать дашборд. Это также лучше соответствует текущей архитектуре экрана, упрощает повторное использование существующих карточек `MealSlotCard` и одновременно закладывает общий execution shell для будущих агентных доменов.

**Alternatives considered:**

- Отдельный full-screen flow. Отклонено: теряется контекст дня и становится сложнее связывать шаги с карточками.
- Обычный modal/sheet. Отклонено: неудобно для поэтапного взаимодействия с карточками на основном экране.
- Несколько параллельных agent sessions. Отклонено: слишком сложно для ментальной модели и UI.

### 3. Общий shell имеет единый lifecycle, а resolve живёт внутри step state

**Decision:** Общий agent shell использует единый lifecycle:

- `idle`
- `chat-input`
- `planning`
- `executing`
- `completed`
- `cancelled`

`resolve` не поднимается в отдельное состояние всего shell-а. Вместо этого это под-состояние активного шага внутри `executing`.

Для шага meal domain используются локальные состояния вроде:

- `draft`
- `needs-resolve`
- `ready-for-confirm`
- `committed`
- `failed`

Planning-phase остаётся внутри chat/composer state. Полноценный execution shell показывается только после успешного результата planner-а.

**Why:** Такой lifecycle переиспользуем между доменами и не привязываем общий shell к meal-специфичной конфликтной логике.

**Alternatives considered:**

- Делать `resolve` отдельным глобальным состоянием shell-а. Отклонено: слишком сильно привязывает общий lifecycle к частным проблемам шага.
- Показывать execution shell уже на этапе planning. Отклонено: пока это лишняя сложность для V1.

### 4. Planner создаёт только обоснованные шаги и сортирует их детерминированно

**Decision:** Planner может вернуть один или несколько шагов. Он создаёт только те шаги, которые может обосновать по сообщению пользователя. Он не должен автоматически форсить полный шаблон `завтрак / обед / ужин`, если в тексте нет достаточных оснований.

Если шагов несколько:

- они группируются по дням;
- порядок групп по дням детерминированный;
- внутри дня шаги идут по порядку meal slots.

Для V1 planner публично поддерживает только чистые сценарии:

- `meal-create`
- `meal-edit`

Смешанные create/edit цепочки остаются вне scope V1, даже если архитектура позже позволит их добавить.

**Why:** Это даёт понятный progress UX и не заставляет систему выдумывать лишние meal slots.

**Alternatives considered:**

- Сохранять порядок как в исходном сообщении пользователя. Отклонено: плохо подходит под guided operational flow.
- Всегда создавать полный шаблон дня. Отклонено: это создаёт ложные шаги.

### 5. Meal draft существует отдельно от persisted `MealEvent`

**Decision:** До подтверждения шага все данные живут только в ephemeral draft state текущей execution chain. После подтверждения:

- для `create` создаётся новый `MealEvent` или выполняется resolve, если слот уже занят;
- для `edit` обновляется уже существующий `MealEvent`;
- затем запускается существующий pipeline сохранения `EstimateItem`, totals и rule recomputation для затронутого дня.

Никакие draft chains не сохраняются в SwiftData. Persist происходят только на meal-level commit.

**Why:** Это поддерживает модель partial completion без resume и не требует сохранять промежуточные agent artifacts в локальном хранилище.

**Alternatives considered:**

- Persist temporary chain state в SwiftData. Отклонено: слишком дорого по сложности для V1.
- Создавать `MealEvent` заранее и потом держать его в dirty-state до подтверждения. Отклонено: загрязнит аналитику и пересчёты.

### 6. После планирования пользователь редактирует только продукты и порции

**Decision:** После того как planner выбрал `date` и `mealSlot`, эти аргументы становятся read-only. Внутри execution mode пользователь может выполнять только локальные действия:

- `add product`
- `remove product`
- `change portion`

Эти действия выполняются детерминированно в локальном UI/runtime state без новых LLM-вызовов.

Если шаг ошибочен по `date` или `mealSlot`, пользователь не исправляет это вручную внутри карточки. Он либо проходит resolve-сценарий, предложенный runtime, либо отменяет оставшуюся цепочку и запускает новую.

**Why:** Это резко снижает сложность chain replanning и делает execution mode предсказуемым.

**Alternatives considered:**

- Разрешить ручное изменение day/slot внутри шага. Отклонено: ломает идентичность шага и создаёт сложные конфликты цепочки.
- Отправлять каждый micro-edit обратно в агент. Отклонено: ухудшает latency и делает базовые правки хрупкими.

### 7. Resolve использует фиксированные UI-контролы для day/slot и агентный draft для products

**Decision:** Нехватка или конфликт в `date` и `mealSlot` разрешаются управляемым UI:

- `date` через быстрые чипсы и календарь;
- `mealSlot` через фиксированные чипсы `завтрак / обед / ужин`;
- переход `create -> edit` через отдельный resolve action.

При этом `productsDraft` остаётся зоной агента/planner-а: в execution mode карточка получает уже нормализованный список продуктов с порциями, который пользователь может локально уточнить.

**Why:** Day/slot являются дискретными аргументами, для которых UI надёжнее агента. Продукты и их разложение по шагам, наоборот, дают основную ценность agent-first подхода.

**Alternatives considered:**

- Полностью отдавать resolve-опции агенту. Отклонено: day/slot слишком просто и надёжно контролируются UI.
- Полностью детерминированно раскладывать продукты без LLM участия. Отклонено: теряется ключевая ценность фичи.

### 8. Конфликт в create не делает silent overwrite

**Decision:** Если в `meal-create` planner/runtime доходит до шага `create(date, mealSlot, ...)`, но для этого `date + mealSlot` уже есть существующий `MealEvent`, шаг переходит в resolve-state. Пользователю предлагается понятный конфликтный сценарий вместо скрытого перезаписывания. Одним из исходов resolve может быть конвертация шага в `edit`.

**Why:** Пользовательский журнал питания нельзя тихо портить. Это особенно важно в продукте про осознанность и самоконтроль.

**Alternatives considered:**

- Автоматически переключать create в edit. Отклонено: слишком неявно.
- Silent overwrite. Отклонено: высокорисковое поведение.

## Risks / Trade-offs

- **[Risk] Planner может неправильно разложить продукты по meal slots.**  
  **Mitigation:** execution mode даёт пользователю возможность проверить и локально поправить состав до commit-а.

- **[Risk] Полное удаление старого slot-based flow повышает стоимость ошибки planner-а.**  
  **Mitigation:** жёстко ограничить V1 двумя сценариями, read-only day/slot после планирования и безопасными resolve-сценариями.

- **[Risk] Multi-day цепочки могут быть когнитивно тяжёлыми.**  
  **Mitigation:** группировать шаги по дням, переключать день только на границе day group и ясно показывать прогресс.

- **[Risk] Отсутствие resume может раздражать пользователя при случайном закрытии mode.**  
  **Mitigation:** явный exit flow с предупреждением о потере неподтверждённого хвоста.

- **[Risk] Новый orchestration слой размажет ответственность между UI и сервисами.**  
  **Mitigation:** зафиксировать отдельные роли classifier, planner и execution runtime, не смешивая их в `DashboardView`.

- **[Risk] Общий agent foundation может быть переусложнён слишком рано.**  
  **Mitigation:** V1 реализует только meal domain, но общие абстракции вводятся только там, где уже есть подтверждённая общая логика: router, shell, lifecycle, single active session.

## Migration Plan

1. Добавить общий agent entry point, router и execution shell без изменения persisted meal models.
2. Встроить глобальную CTA-кнопку поверх `ContentView`/`TabView`.
3. Реализовать meal domain как первый classifier/planner/renderer внутри общего agent foundation.
4. Перевести create/edit meal logging на новый agent-first flow на `DashboardView`.
5. Удалить старый slot-based ввод и прямой сценарий проваливания в meal input/editor как основной способ логирования.
6. Сохранить существующий pipeline сохранения `MealEvent`, `EstimateItem` и rule recomputation после meal-level commit.
7. Обновить тесты на сценарии create/edit, conflicts, partial completion, multi-day transitions и single active session.

Rollback упрощён тем, что change не требует миграции persisted данных: основное изменение находится в orchestration и UI flow.

- Нужно ли в V1 визуально разрешать переход по уже завершённым шагам progress bar или он будет только индикатором текущего выполнения.
