## Why

Сейчас ввод приёмов пищи завязан на отдельные слоты завтрака, обеда и ужина, из-за чего пользователь вынужден заходить внутрь конкретной карточки и заполнять данные в локальном контексте этого слота. Такой поток создаёт лишнюю навигацию и плохо подходит для более естественного agent-first сценария, в котором пользователь свободно описывает еду или правку, а система сама распознаёт сценарий, строит план действий и подготавливает большую часть данных к подтверждению человеком.

## What Changes

- Добавить основную кнопку ввода по центру нижней части экрана как глобальную точку входа для логирования и правки приёмов пищи.
- Открывать по этой кнопке свободный чат ввода, который становится основным пользовательским сценарием создания и изменения записей о еде.
- Убрать старый slot-based flow ввода через проваливание в карточку конкретного приёма пищи и заменить его единым smart input flow.
- Перенести часть информации, которая сейчас доступна только внутри экрана приёма пищи, обратно на карточку приёма на дашборде, чтобы execution flow происходил на основном экране.
- Ввести новый pipeline обработки сообщения: `scenario classification -> plan building -> execution mode`.
- На первом этапе поддержать только два сценария: `meal-create` и `meal-edit`.
- После классификации строить цепочку нормализованных шагов `create(date, mealSlot, products)` или `edit(date, mealSlot, products)` вместо немедленного сохранения данных.
- Разрешить multi-step и multi-day цепочки, если planner может обоснованно разложить сообщение на несколько шагов; шаги должны группироваться по дням и идти в детерминированном порядке внутри дня.
- Ввести execution mode на дашборде: после построения плана чат скрывается, внизу появляется progress bar шагов с возможностью отмены всей оставшейся цепочки через явный exit flow.
- Для каждого шага агент должен предзаполнить карточку приёма пищи нормализованным draft-списком продуктов и порций, после чего пользователь подтверждает шаг или локально редактирует только состав и порции.
- Зафиксировать, что после планирования `date` и `mealSlot` становятся read-only для пользователя; проблемы в этих аргументах разрешаются только через resolve-сценарий, а не прямым ручным редактированием шага.
- Разрешить в карточке шага только детерминированные локальные действия `add product`, `remove product`, `change portion` без повторного вызова агента на каждое микродействие.
- Передавать в LLM существующий контекст дня, если запрос относится к конкретному дню и это помогает корректно построить шаги создания или редактирования.
- При конфликте в сценарии создания, если запись для `date + mealSlot` уже существует, не делать молчаливый overwrite, а переводить шаг в resolve-состояние с возможностью перейти к редактированию или отменить/переформулировать шаг.
- Сохранять каждый подтверждённый шаг отдельно: после подтверждения draft превращается в реальный `MealEvent`, запускается обычный pipeline оценки и пересчёта правил только для затронутого дня.
- Сделать execution chain одноразовой: подтверждённые шаги сохраняются, неподтверждённый хвост теряется при отмене, восстановление прерванной цепочки не поддерживается.

## Capabilities

### New Capabilities
- `chat-based-meal-logging`: глобальная точка входа в логирование питания через нижнюю CTA-кнопку и свободный chat-first сценарий ввода.
- `meal-scenario-classification`: распознавание пользовательского сценария `meal-create` или `meal-edit`, а также определение релевантного временного контекста.
- `meal-step-planning`: преобразование свободного сообщения в цепочку нормализованных шагов `create/edit` с day grouping, meal targeting и draft payload для исполнения.
- `meal-execution-mode`: execution flow на дашборде с progress bar, поэтапным подтверждением шагов, resolve-состояниями и локальным редактированием draft-состава приёма пищи.

### Modified Capabilities
- `event-logging`: логирование приёмов пищи меняет основной пользовательский поток с slot-based ввода на agent-prepared execution flow, где система строит plan chain, предзаполняет meal drafts и сохраняет записи только после поэтапного подтверждения.
- `llm-agent`: LLM должен участвовать не только в оценке питания, но и в scenario classification, plan building, meal draft preparation и использовании контекста существующего дня при релевантных запросах.

## Impact

- Затронутые экраны: `ContentView`, `DashboardView`, карточки приёмов пищи, нижняя CTA-зона, текущий flow ввода и открытия meal details.
- Затронутые сервисы: orchestration слой для scenario classification и plan building, `LLMClient`, prompt building, runtime chain state, pipeline сохранения и обновления `MealEvent`.
- Затронутые данные и поведение: маршрутизация пользовательского ввода, построение multi-step/multi-day цепочек, draft state до подтверждения, использование day context при create/edit сценариях.
- Потребуются новые spec-артефакты для scenario classification, step planning и execution mode, а также изменения существующих требований для event logging и LLM interaction.
