## ADDED Requirements

### Requirement: Agent-prepared подтверждение перед сохранением meal event
Система SHALL сохранять meal event только после подтверждения пользователем подготовленного meal draft-а внутри execution mode.

#### Scenario: Draft не подтверждён
- **WHEN** planner уже подготовил шаг, но пользователь ещё не подтвердил meal draft
- **THEN** система не создаёт и не обновляет persisted `MealEvent`

#### Scenario: Draft подтверждён
- **WHEN** пользователь подтверждает активный meal draft
- **THEN** система создаёт новый `MealEvent` или обновляет существующий в зависимости от типа шага

### Requirement: Независимый meal-level commit внутри цепочки
Система SHALL коммитить каждый подтверждённый шаг независимо от остальных шагов той же цепочки.

#### Scenario: Частичное завершение цепочки
- **WHEN** пользователь подтвердил первые шаги цепочки и затем отменил execution mode
- **THEN** подтверждённые `MealEvent` сохраняются, а неподтверждённый хвост не создаётся

#### Scenario: Коммит одного шага не ждёт завершения всей цепочки
- **WHEN** пользователь подтверждает отдельный meal step
- **THEN** этот шаг сохраняется сразу, без ожидания окончания всей execution chain

### Requirement: Немедленный пересчёт после meal-level commit
Система SHALL после каждого подтверждённого meal-level commit запускать обычный pipeline оценки и пересчёта для затронутого дня.

#### Scenario: Подтверждение шага создания
- **WHEN** подтверждён шаг `create`
- **THEN** система сохраняет событие, создаёт/обновляет items и пересчитывает правила для соответствующего дня

#### Scenario: Подтверждение шага редактирования
- **WHEN** подтверждён шаг `edit`
- **THEN** система обновляет событие, пересчитывает структуру и обновляет rule state для соответствующего дня

### Requirement: Конфликт create-сценария не перезаписывает существующий meal event
Система SHALL переводить `create`-шаг в resolve-state, если для `date + mealSlot` уже существует запись, и MUST NOT молча перезаписывать её.

#### Scenario: Слот уже занят
- **WHEN** runtime исполняет шаг `create(date, mealSlot, ...)` и на этот `date + mealSlot` уже существует `MealEvent`
- **THEN** шаг переходит в resolve-state вместо автоматического overwrite

#### Scenario: Resolve переводит create в edit
- **WHEN** пользователь подтверждает resolve-вариант перехода к редактированию существующего слота
- **THEN** система продолжает исполнение шага как `edit`, а не как silent overwrite
