## ADDED Requirements

### Requirement: LLM участвует в meal scenario classification
Система SHALL использовать LLM для распознавания meal-сценария и релевантного временного контекста до построения шагов.

#### Scenario: Распознавание create-сценария
- **WHEN** пользователь отправляет свободное meal-сообщение без явной команды редактирования
- **THEN** LLM-классификация возвращает сценарий `meal-create`

#### Scenario: Распознавание edit-сценария
- **WHEN** пользователь отправляет сообщение с намерением изменить уже существующую meal-запись
- **THEN** LLM-классификация возвращает сценарий `meal-edit`

### Requirement: LLM участвует в meal step planning
Система SHALL использовать LLM для построения нормализованного meal plan-а с шагами и draft payload-ами для execution mode.

#### Scenario: Один запрос преобразуется в несколько meal steps
- **WHEN** сообщение пользователя относится к нескольким приёмам пищи
- **THEN** LLM/planner возвращает цепочку нескольких шагов с указанием `date`, `mealSlot` и `productsDraft`

#### Scenario: Draft продуктов для шага
- **WHEN** LLM/planner возвращает meal step
- **THEN** шаг содержит нормализованный draft списка продуктов с порциями для предзаполнения карточки

### Requirement: Контекст существующего дня передаётся в релевантных meal-сценариях
Система SHALL передавать в LLM контекст существующего дня, если это помогает корректно построить create/edit шаги для этого дня.

#### Scenario: Редактирование существующего дня
- **WHEN** meal-сообщение относится к дню, в котором уже есть сохранённые приёмы пищи
- **THEN** LLM получает контекст существующего дня до построения итогового плана

#### Scenario: Create-конфликт на занятом слоте
- **WHEN** система строит create-сценарий для дня, в котором уже есть запись в соответствующем слоте
- **THEN** LLM/planner может использовать day context для подготовки resolve-варианта вместо blind create
