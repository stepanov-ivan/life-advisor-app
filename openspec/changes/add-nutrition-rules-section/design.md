## Context

Приложение Life Advisor — iOS-приложение на SwiftUI + SwiftData для трекинга питания. Пользователь описывает приёмы пищи свободным текстом, LLM (DeepSeek/OpenAI) парсит описание в структурированные данные: калории, БЖУ, ингредиенты с граммовкой. Данные хранятся локально в SQLite через SwiftData.

Текущая аналитика (`AnalyticsView`) — чисто агрегативная: суммы калорий, распределение БЖУ, графики за день/неделю/месяц. Нет оценки качества рациона относительно стандартов, нет паттерновой обратной связи.

Цель дизайна — спроектировать rule engine, который оценивает питание пользователя относительно 16 правил ВОЗ/ФАО (2024) и показывает нарушения с drill-down до конкретного ингредиента.

**Ограничения:**
- SwiftData без миграций (destructive reset при изменении схемы)
- LLM нестабилен: оценка может вернуть `parseFailed`, confidence может быть низкой
- Приложение однопользовательское, без облачной синхронизации
- Все вычисления правил — чистая арифметика, без вызовов LLM
- iOS 17+ (SwiftData доступен)

## Goals / Non-Goals

**Goals:**
- 16 правил ВОЗ/ФАО в JSON-конфигурации, расширяемой без перекомпиляции логики примитивов
- Единый вычислительный примитив `range` с опциональными границами и параметром окна (`day`/`week`), покрывающий все 16 правил
- Трёхзонная система оценки: normal / warning (≥85% порога) / violation
- Фиксированная неделя (понедельник–воскресенье) с минимумом 3 дней данных
- Инкрементальный пересчёт: первичный слепок → дельта нового дня
- Статус правила по today (daily freshness): пользователь видит состояние на сегодня; если данных за сегодня нет — «нет данных»
- Drill-down трассировка: правило → день → приём → ингредиент
- Навигация от нарушения к приёму в дашборде: `selectedDate` поднят на уровень `ContentView`, проброшен как `@Binding`
- Подсветка нарушений в существующей ленте питания (MealSlotCard, MealEventEditorView): бордер по худшей зоне, бейдж с количеством нарушений
- MealSlotCard вынесен в отдельный файл при рефакторинге DashboardView

**Non-Goals:**
- Конфигурация порогов пользователем (пороги ВОЗ фиксированы)
- Локальный классификатор продуктов (полагаемся на LLM foodCategory)
- Правило «Вода» (нет ввода потребления воды)
- Period switcher на графике детали (только текущая неделя)
- Tracking mode (отдельное от enabled/disabled состояние)
- Миграция существующих приёмов для заполнения новых полей (старые выпадают естественно)
- Сигналы гипотез (fish/dairy avoidance) — не расширяем, делаем минимально по ВОЗ

## Decisions

### 1. JSON-driven правила вместо хардкод-enum

**Выбор:** Правила определены в `nutrition_rules.json` в bundle. `RuleDefinition` — Codable struct. `RuleEngine` загружает JSON, интерпретирует, вычисляет. `NutritionRuleConfig` в SwiftData хранит только `ruleId` и `isEnabled` — пользовательские настройки.

**Альтернативы:**
- Enum с фиксированными case: добавление нового правила = изменение кода + перекомпиляция. Отклонено — требование расширяемости.
- Data-driven из JSON: легко добавить правило без изменения evaluation-логики (если примитив уже существует). Выбрано.

**Структура JSON:**
```json
{
  "id": "who_total_fat",
  "type": "range",
  "field": "fatPercent",
  "params": { "lower": 0.15, "upper": 0.30 },
  "warningRatio": 0.85,
  "category": "macronutrient_balance",
  "title": "Жиры в норме",
  "description": "ВОЗ: 15–30% суточной энергии",
  "window": "day",
  "minDaysData": 3
}
```

Ключевые поля:
- `type` — всегда `"range"` (единственный примитив)
- `field` — имя вычисляемого поля (fatPercent, proteinPercent, fiberGrams, ...)
- `params` — `lower` и/или `upper` (оба опциональны; отсутствие границы = нет ограничения с этой стороны)
- `window` — `"day"` или `"week"`
- `category` — для группировки в UI (macronutrient_balance, restrictions, food_quality, eating_pattern)
- `warningRatio` — порог warning-зоны (0.85 = 85% от границы)

**Валидация при загрузке:** `RuleEngine.loadRules()` проверяет, что `field` соответствует фактическим вычисляемым полям модели, `type` равен `"range"`, `window` — `"day"` или `"week"`, границы корректны. Некорректные правила логируются и пропускаются.

### 2. Единый примитив range вместо 4 разных

**Выбор:** Один вычислительный примитив `range` с опциональными `lower`/`upper` и параметром `window: "day" | "week"`.

**Было в исходном дизайне:** 4 примитива (range, max, min, maxWeekly). `max` — вырожденный `range` с `lower = -inf`, `min` — `range` с `upper = +inf`, `maxWeekly` — `range` с окном в 7 дней. Разделение избыточно, warning-зона считается одинаково для всех вариантов.

**Примеры:**
```json
// Белок 10–15% — обе границы
{ "type": "range", "params": { "lower": 0.10, "upper": 0.15 }, "window": "day" }

// Сахар ≤10% — только верхняя граница
{ "type": "range", "params": { "upper": 0.10 }, "window": "day" }

// Клетчатка ≥25g — только нижняя граница
{ "type": "range", "params": { "lower": 25 }, "window": "day" }

// Красное мясо ≤500g/нед — недельное окно
{ "type": "range", "params": { "upper": 500 }, "window": "week" }
```

**Файлы:**
- `Services/RuleEngine.swift` — оркестрация: загрузка JSON, enabled-фильтр, вызов примитива, зоны, инкремент
- `Services/RuleEvaluation.swift` — чистая функция range-примитива

### 3. Новые поля только в EstimateItem, агрегаты на лету

**Выбор:** `estimatedSaturatedFats`, `estimatedSugar`, `estimatedFiber`, `estimatedSodium`, `foodCategory` добавляются только в `EstimateItem`. При оценке правил агрегаты суммируются reduce(+) по items приёма.

**Альтернативы:**
- Дублировать в MealEvent + EstimateItem (как calories/BJU сейчас): риск рассинхронизации, два источника истины. Отклонено.
- Только в EstimateItem: один источник, агрегаты всегда консистентны, вычисление дёшево (≤5 items × ≤5 приёмов = 25 операций). Выбрано.

Существующее дублирование calories/BJU в `MealEvent` — исторически сложившийся паттерн, новые поля его не продолжают.

Для presence-правил и категорийных min/max — движок фильтрует items по `foodCategory` и суммирует/проверяет.

### 4. Опциональные поля в LLM-контракте

**Выбор:** Все новые поля (`saturated_fats`, `sugar`, `fiber`, `sodium`, `food_category`) — опциональны в JSON-ответе LLM. Отсутствие поля не ломает `parseFailed` — правило получает «недостаточно данных». Один промпт для всех полей, без разделения на базовый/расширенный запросы.

**Альтернативы:**
- Обязательные поля: больше отказов при парсинге, потеря всего приёма из-за отсутствия `sodium`. Отклонено.
- Два отдельных запроса: овер-инжиниринг, сильные LLM справляются с одним промптом.
- Опциональные: приём сохраняется, правила постепенно наполняются данными по мере улучшения LLM. Выбрано.

**Список foodCategory (8 значений):** `fruit`, `vegetable`, `whole_grain`, `legume`, `nut_seed`, `red_meat`, `processed_meat`, `other`. Только категории, нужные правилам ВОЗ. Минимизирует галлюцинации.

### 5. Инкрементальный пересчёт

**Выбор:** При первом запуске (или входе на экран правил) вычисляется полный слепок за текущую неделю. При сохранении нового приёма — пересчитываются только нарушения для этого дня. При редактировании старого приёма — пересчитывается только его день. При наступлении понедельника — старые `RuleViolation` удаляются.

**Альтернативы:**
- Полный пересчёт при каждом сохранении: избыточно. Хотя данных мало, инкремент — чище архитектурно.
- Инкремент: слепок один раз, дельта daily. Выбрано.

**Хранение:** `RuleViolation` сохраняется в SwiftData для исторического графика в пределах текущей недели. При сбросе недели старые записи удаляются.

### 6. Daily freshness для статуса правила

**Выбор:** Индикатор на экране списка правил показывает статус за сегодня. Если сегодня данных ещё нет (утро, ни одного приёма) — статус «нет данных» (серый). Если сегодня жиры 34% — правило красное, даже если вчера было 28%.

**Альтернативы:**
- Оконный статус: правило помнит нарушения всю неделю. Нельзя «исправить» правило сегодня — демотивирует. Отклонено.
- Daily freshness: сегодня чисто → зелёный. Даёт ощущение контроля. Окно используется только для графика и списка нарушений на экране детали. Выбрано.

### 7. Три зоны оценки

**Выбор:** normal / warning / violation. Warning = ≥85% порога (warningRatio в JSON).

**Альтернативы:**
- Бинарно (нарушено/нет): неинформативно.
- Continuous score (0–100): сложно для пользователя, избыточно для v1.
- Три зоны: интуитивно (зелёный/жёлтый/красный), легко реализовать. Выбрано.

Зоны для range с двумя границами — warning с обеих сторон. Для range с одной границей — warning только со стороны ограничения.

### 8. Фиксированная неделя (пн–вс) вместо скользящего окна

**Выбор:** Неделя с понедельника по воскресенье. В понедельник недельные правила сбрасываются, старые `RuleViolation` удаляются. Частичная неделя (меньше 7 дней) оценивается при наличии minDaysData=3 дней.

**Было в исходном дизайне:** Скользящее 7-дневное окно. Пользователь спросил — «может скользящее окно хрень, давай зафиксируем по неделям». Выбрано: проще для ментальной модели пользователя, сброс в воскресенье/понедельник интуитивен.

**Поведение при сбросе:**
- Weekly-правила: значения обнуляются, старые `RuleViolation` удаляются из SwiftData
- Day-правила: без изменений, daily freshness как обычно
- График на детали правила показывает только текущую неделю

### 9. Присутствие категории через any match за день

**Выбор:** Для presence-правил (`who_whole_grain`, `who_legumes_nuts`, `who_processed_meat`) — день считается satisfied, если хотя бы один ингредиент за день имеет нужную `foodCategory`.

**Альтернативы:**
- Каждый приём должен содержать категорию: слишком жёстко, не соответствует духу ВОЗ.
- Majority (≥2 приёмов): произвольный порог.
- Any match: «включать цельные злаки в рацион ежедневно» = хотя бы одна порция за день. Выбрано.

### 10. Skipped и parseFailed в оценке

**Выбор:** Skipped-приёмы исключаются из всех правил кроме `who_meal_regularity`. ParseFailed-приёмы исключаются из оценки полностью (missing data). Если **все** приёмы за день skipped — день пропускается в агрегации, minDaysData уменьшается.

**Альтернативы:**
- Skipped = 0 калорий: пользователь осознанно пропустил приём (интервальное голодание, разгрузочный день), не должен получить violation «недобор белка». Отклонено.
- Skipped исключается из калорийных/макро-правил. Правило регулярности само следит за частотой пропусков.

### 11. Навигация от нарушения к приёму — selectedDate на ContentView

**Выбор:** `selectedDate` поднимается на уровень `ContentView` (где лежит `TabView`), пробрасывается как `@Binding` в `DashboardView`. Тап по нарушению на экране детали правила → `RuleDetailView` пишет дату в тот же source of truth → программное переключение таба на Дашборд → открытие `MealEventEditorView`.

**Альтернативы:**
- @AppStorage для навигации: костыль, @AppStorage не для UI-состояния.
- Синглтон: грязно, тестировать сложно.
- Подъём на ContentView: чистый SwiftUI-паттерн, один источник истины. Выбрано.

### 12. Миграция — destructive reset

**Выбор:** Использовать существующий механизм сброса схемы. После обновления пользователь начинает с чистой базы.

**Альтернативы:**
- SwiftData SchemaMigrationPlan: сохраняет данные, но сложен (особенно без опыта миграций в проекте). Для v1 с ограниченной аудиторией destructive reset допустим.
- Выбрано: destructive reset. Пользователи теряют историю, правила наполняются за 7 дней.

### 13. Архитектура файлов

```
Models/
  NutritionRuleConfig.swift    // @Model: ruleId, isEnabled
  RuleViolation.swift          // @Model: ruleId, date, zone, magnitude, reasonCode
                                // @Relationship mealEvent: MealEvent?
                                // @Relationship estimateItem: EstimateItem?

Services/
  RuleEngine.swift             // Оркестрация: loadRules, evaluate, incrementalEvaluate, zone, violationsForMeal
  RuleEvaluation.swift         // Чистая функция range-примитива

Views/
  RulesListView.swift          // Список правил с 4 секциями
  RuleDetailView.swift         // Деталь правила: график + список нарушений
  MealSlotCard.swift           // Вынесен из DashboardView при рефакторинге

Resources/
  nutrition_rules.json         // 16 правил ВОЗ
```

Правила не имеют собственного SwiftData-хранилища (кроме `NutritionRuleConfig` для enabled/disabled). Определения правил читаются из JSON. Это позволяет обновлять пороги без изменения кода.

### 14. Интеграция с DashboardView

DashboardView при `onAppear` и после `saveMealEvent` вызывает `RuleEngine.violationsForMeal(meal:)` для каждого приёма дня. Результат передаётся в `MealSlotCard`:
- **Бордер**: цвет по худшей зоне (если хоть одно violation — красный)
- **Бейдж**: количество нарушений (число)

MealEventEditorView при открытии вызывает `RuleEngine.violationsForItems(meal:)` для списка ингредиентов — помечает строки-нарушители иконкой.

### 15. Правило energy_balance

**Выбор:** Пользователь вводит целевую калорийность в настройках. Правило сравнивает фактическое потребление с целью через `frequency`-логику (дней превышения из 7). Реализуется через тот же range-примитив с `window: "week"`.

### 16. Проценты от общей калорийности

Для макро-правил проценты считаются от общей калорийности за день по стандартным коэффициентам:
- Жиры: `(fatGrams × 9) / totalCalories`
- Белки: `(proteinGrams × 4) / totalCalories`  
- Сахар: `(sugarGrams × 4) / totalCalories`

Если `totalCalories = 0` (все приёмы skipped или нет данных) — правило получает «нет данных» за этот день.

## Risks / Trade-offs

- **[Risk] LLM не возвращает новые поля → правила показывают «недостаточно данных».** Mitigation: поля опциональны, правила постепенно наполняются по мере стабилизации LLM. Не ломает существующий flow.
- **[Risk] Destructive reset стирает историю → правила пусты первые 3–7 дней.** Mitigation: принято осознанно (Non-Goal: миграция). Пользователь видит прогресс-бар «N из 3 дней данных».
- **[Risk] Отсутствие `foodCategory` у ингредиентов → presence-правила не оцениваются.** Mitigation: 8 значений в промпте LLM. Если категория не указана — `"other"`, правило получает missing data.
- **[Risk] JSON правила могут быть некорректны (опечатка в field name) → правило молча не работает.** Mitigation: валидация JSON при загрузке в `RuleEngine.loadRules()` — проверка field names на соответствие модели, type == "range", window в ["day", "week"]. Некорректные правила логируются и пропускаются.
- **[Risk] Единый примитив range не покрывает presence и countSkipped.** Эти два правила (`who_whole_grain`, `who_meal_regularity`) — особые случаи, не вписываются в range. Реализуются отдельными проверками в `RuleEngine` без выделения в полноценный примитив.
- **[Trade-off] Инкрементальный пересчёт сложнее полного.** Выигрыш в производительности для недельного окна незначителен. Но архитектура чище: слепок один раз, дельта daily. При баге — полный пересчёт всегда доступен как fallback.
- **[Trade-off] Навигация через переключение табов теряет контекст на дашборде.** Пользователь может быть на другом дне. Принято — компромисс в пользу естественного поведения «покажи тот приём».
- **[Trade-off] Фиксированная неделя проще скользящего окна, но в понедельник данные сбрасываются.** Пользователь не видит ретроспективу дальше текущей недели. Принято — для v1 достаточно, в будущем можно добавить историю.
