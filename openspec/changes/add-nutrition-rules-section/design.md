## Context

Приложение Life Advisor — iOS-приложение на SwiftUI + SwiftData для трекинга питания. Пользователь описывает приёмы пищи свободным текстом, LLM (DeepSeek/OpenAI) парсит описание в структурированные данные: калории, БЖУ, ингредиенты с граммовкой. Данные хранятся локально в SQLite через SwiftData.

Текущая аналитика (`AnalyticsView`) — чисто агрегативная: суммы калорий, распределение БЖУ, графики за день/неделю/месяц. Нет оценки качества рациона относительно стандартов, нет паттерновой обратной связи.

Цель дизайна — спроектировать rule engine, который оценивает питание пользователя относительно 16 правил ВОЗ/ФАО (2024) и показывает нарушения с drill-down до конкретного ингредиента.

**Ограничения:**
- SwiftData без миграций (destructive reset при изменении схемы)
- LLM нестабилен: оценка может вернуть `parseFailed`, confidence может быть низкой
- Приложение однопользовательское, без облачной синхронизации
- Все вычисления правил — чистая арифметика, без вызовов LLM
- iOS 17+ (SwiftData доступен)

## Goals / Non-Goals

**Goals:**
- 16 правил ВОЗ/ФАО в JSON-конфигурации, расширяемой без перекомпиляции логики примитивов
- 7 вычислительных примитивов, покрывающих все 16 правил
- Трёхзонная система оценки: normal / warning (≥85% порога) / violation
- Скользящее 7-дневное окно с минимумом 3 дней данных
- Инкрементальный пересчёт: первичный слепок → дельта нового дня
- Статус правила по today (daily freshness): пользователь видит состояние на сегодня
- Drill-down трассировка: правило → день → приём → ингредиент
- Навигация от нарушения к приёму в дашборде
- Подсветка нарушений в существующей ленте питания (MealSlotCard, MealEventEditorView)

**Non-Goals:**
- Конфигурация порогов пользователем (пороги ВОЗ фиксированы)
- Локальный классификатор продуктов (полагаемся на LLM foodCategory)
- Правило «Вода» (нет ввода потребления воды)
- Period switcher на графике детали (только 7 дней)
- Tracking mode (отдельное от enabled/disabled состояние)
- Миграция существующих приёмов для заполнения новых полей (старые выпадают естественно)

## Decisions

### 1. JSON-driven правила вместо хардкод-enum

**Выбор:** Правила определены в `nutrition_rules.json` в bundle. `RuleDefinition` — Codable struct. `RuleEngine` загружает JSON, интерпретирует, вычисляет.

**Альтернативы:**
- Enum с фиксированными case: добавление нового правила = изменение кода + перекомпиляция. Отклонено — требование расширяемости.
- Data-driven из JSON: легко добавить правило без изменения evaluation-логики (если примитив уже существует). Выбрано.

**Структура JSON:**
```json
{
  "id": "who_total_fat",
  "type": "range",
  "field": "fatPercent",
  "params": { "lower": 0.15, "upper": 0.30 },
  "warningRatio": 0.85,
  "category": "macronutrient_balance",
  "title": "Жиры в норме",
  "description": "ВОЗ: 15–30% суточной энергии",
  "window": "day",
  "minDaysData": 3
}
```

Ключевые поля:
- `type` — один из 7 примитивов (range, max, min, maxWeekly, presence, countSkipped, frequency)
- `field` — имя вычисляемого поля (fatPercent, proteinPercent, fiberGrams, ...)
- `params` — специфичные для типа параметры (lower/upper для range, threshold для max/min, categories для агрегации)
- `category` — для группировки в UI (macronutrient_balance, restrictions, food_quality, eating_pattern)

### 2. 7 примитивов вместо expression-based движка

**Выбор:** Фиксированный набор из 7 вычислительных примитивов, каждый реализован как чистая функция в `RuleEvaluation`.

**Альтернативы:**
- Expression-based DSL с парсером: мощно, но требует языка выражений, парсера, валидации, сложного UI. Овер-инжиниринг для 16 правил.
- 7 примитивов: каждый покрывает 1–4 правила. Добавление нового примитива = ~20 строк кода. Выбрано.

**Состав примитивов:**

| Примитив | Сигнатура | Пример |
|---|---|---|
| `range` | value ∈ [lower, upper] | Белок 10–15% |
| `max` | value ≤ threshold | Сахар ≤10% |
| `min` | value ≥ threshold | Клетчатка ≥25g |
| `maxWeekly` | sum(7d) ≤ threshold | Красное мясо ≤500g/нед |
| `presence` | any item matches category | Цельные злаки есть |
| `countSkipped` | подсчёт empty-окон | Регулярность |
| `frequency` | дней превышения из 7 | Энергобаланс |

**Файлы:**
- `Services/RuleEngine.swift` — оркестрация: загрузка JSON, enabled-фильтр, вызов примитивов, зоны, инкремент
- `Services/RuleEvaluation.swift` — чистые функции для 7 примитивов

### 3. Новые поля только в EstimateItem, агрегаты на лету

**Выбор:** `estimatedSaturatedFats`, `estimatedSugar`, `estimatedFiber`, `estimatedSodium`, `foodCategory` добавляются только в `EstimateItem`. `MealEvent` не хранит агрегаты новых полей. При оценке правил агрегаты суммируются reduce(+) по items приёма.

**Альтернативы:**
- Дублировать в MealEvent + EstimateItem (как calories/BJU сейчас): риск рассинхронизации, два источника истины. Отклонено.
- Только в EstimateItem: один источник, агрегаты всегда консистентны, вычисление дёшево (≤5 items × ≤5 приёмов = 25 операций). Выбрано.

Для presence-правил и категорийных min/max — движок фильтрует items по `foodCategory` и суммирует/проверяет.

### 4. Опциональные поля в LLM-контракте

**Выбор:** Все новые поля (`saturated_fats`, `sugar`, `fiber`, `sodium`, `food_category`) — опциональны в JSON-ответе LLM. Отсутствие поля не ломает `parseFailed` — правило получает «недостаточно данных».

**Альтернативы:**
- Обязательные поля: больше отказов при парсинге, потеря всего приёма из-за отсутствия `sodium`. Отклонено.
- Опциональные: приём сохраняется, правила постепенно наполняются данными по мере улучшения LLM. Выбрано.

**Список foodCategory (8 значений):** `fruit`, `vegetable`, `whole_grain`, `legume`, `nut_seed`, `red_meat`, `processed_meat`, `other`. Только категории, нужные правилам ВОЗ. Минимизирует галлюцинации.

### 5. Инкрементальный пересчёт

**Выбор:** При первом запуске (или входе на экран правил) вычисляется полный слепок за 7 дней. При сохранении нового приёма — пересчитываются только нарушения для этого дня. При редактировании старого приёма — пересчитывается только его день. При сдвиге окна (день 8 выпадает) — удаляются старые `RuleViolation`.

**Альтернативы:**
- Полный пересчёт при каждом сохранении: 7 × 5 × 5 = 175 записей. SwiftData справляется мгновенно, но избыточно.
- Инкремент: честный компромисс — слепок один раз, дельта daily. Выбрано.

**Хранение:** `RuleViolation` сохраняется в SwiftData для исторического графика. При сдвиге окна старые записи удаляются.

### 6. Daily freshness для статуса правила

**Выбор:** Индикатор на экране списка правил показывает статус за сегодня, а не за всё окно. Если сегодня жиры 34% — правило красное, даже если вчера было 28%.

**Альтернативы:**
- Оконный статус: правило помнит нарушения всю неделю. Нельзя «исправить» правило сегодня — демотивирует. Отклонено.
- Daily freshness: сегодня чисто → зелёный. Даёт ощущение контроля. Окно используется только для графика и списка нарушений на экране детали. Выбрано.

### 7. Три зоны оценки

**Выбор:** normal / warning / violation. Warning = ≥85% порога (warningRatio в JSON).

**Альтернативы:**
- Бинарно (нарушено/нет): неинформативно.
- Continuous score (0–100): сложно для пользователя, избыточно для v1.
- Три зоны: интуитивно (зелёный/жёлтый/красный), легко реализовать. Выбрано.

Зоны применимы ко всем примитивам: для range — обе границы имеют warning-зоны с обеих сторон, для max/min — только со стороны превышения.

### 8. Присутствие категории через any match за день

**Выбор:** Для presence-правил (`who_whole_grain`, `who_legumes_nuts`, `who_processed_meat`) — день считается satisfied, если хотя бы один ингредиент за день имеет нужную `foodCategory`.

**Альтернативы:**
- Каждый приём должен содержать категорию: слишком жёстко, не соответствует духу ВОЗ.
- Majority (≥2 приёмов): произвольный порог.
- Any match: «включать цельные злаки в рацион ежедневно» = хотя бы одна порция за день. Выбрано.

### 9. Skipped и parseFailed в оценке

**Выбор:** Skipped-приёмы исключаются из всех правил кроме `who_meal_regularity`. ParseFailed-приёмы исключаются из оценки полностью (missing data).

**Альтернативы:**
- Skipped = 0 калорий: пользователь осознанно пропустил приём (интервальное голодание, разгрузочный день), не должен получить violation «недобор белка». Отклонено.
- Skipped исключается из калорийных/макро-правил: выбран подход B. Правило регулярности само следит за частотой пропусков.

### 10. Навигация от нарушения к приёму — переключение таба

**Выбор:** Тап по нарушению на экране детали правила → программное переключение `TabView` на Дашборд → установка `dashboard_last_selected_day` → открытие `MealEventEditorView` sheet.

**Альтернативы:**
- Модальный показ из правил: дублирование логики открытия, два пути к одному экрану.
- Навигация внутри правил без перехода: пользователь не видит контекст дня.
- Переключение таба: естественное поведение, один источник истины для открытия приёма. Выбрано.

### 11. Миграция — destructive reset

**Выбор:** Использовать существующий механизм сброса схемы. После обновления пользователь начинает с чистой базы.

**Альтернативы:**
- SwiftData SchemaMigrationPlan: сохраняет данные, но сложен (особенно без опыта миграций в проекте). Для v1 с ограниченной аудиторией destructive reset допустим.
- Выбрано: destructive reset. Пользователи теряют историю, правила наполняются за 7 дней.

### 12. Архитектура файлов

```
Models/
  NutritionRuleConfig.swift    // @Model: ruleId, isEnabled
  RuleViolation.swift          // @Model: ruleId, date, zone, magnitude, reasonCode, mealEventId?, estimateItemId?

Services/
  RuleEngine.swift             // Оркестрация: loadRules, evaluate, incrementalEvaluate, zone, violationsForMeal
  RuleEvaluation.swift         // 7 чистых функций-примитивов

Views/
  RulesListView.swift          // Список правил с 4 секциями
  RuleDetailView.swift         // Деталь правила: график + список нарушений

Resources/
  nutrition_rules.json         // 16 правил ВОЗ
```

Правила не имеют собственного SwiftData-хранилища (кроме `NutritionRuleConfig` для enabled/disabled). Определения правил читаются из JSON. Это позволяет обновлять пороги без изменения кода.

### 13. Интеграция с DashboardView

DashboardView при `onAppear` и после `saveMealEvent` вызывает `RuleEngine.violationsForMeal(meal:)` для каждого приёма дня. Результат передаётся в `MealSlotCard` для отрисовки бордера и бейджа.

MealEventEditorView при открытии вызывает `RuleEngine.violationsForItems(meal:)` для списка ингредиентов — помечает строки-нарушители иконкой.

## Risks / Trade-offs

- **[Risk] LLM не возвращает новые поля → правила показывают «недостаточно данных».** Mitigation: поля опциональны, правила постепенно наполняются по мере стабилизации LLM. Не ломает существующий flow.
- **[Risk] Destructive reset стирает историю → правила пусты первые 7 дней.** Mitigation: принято осознанно (Non-Goal: миграция). Пользователь видит прогресс-бар «N из 3 дней данных».
- **[Risk] Отсутствие `foodCategory` у ингредиентов → presence-правила не оцениваются.** Mitigation: 8 значений в промпте LLM. Если категория не указана — `"other"`, правило получает missing data.
- **[Risk] JSON правила могут быть некорректны (опечатка в field name) → правило молча не работает.** Mitigation: валидация JSON при загрузке в `RuleEngine.loadRules()` с логированием ошибок. Некорректные правила пропускаются.
- **[Trade-off] Инкрементальный пересчёт сложнее полного.** Выигрыш в производительности для 7-дневного окна незначителен. Но архитектура готова к полному пересчёту если инкремент станет проблемой.
- **[Trade-off] Навигация через переключение табов теряет контекст на дашборде.** Пользователь может быть на другом дне. Принято — компромисс в пользу естественного поведения «покажи тот приём».
