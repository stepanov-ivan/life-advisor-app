## Context

Текущее приложение строилось вокруг справочника `FoodItem` и structured-ввода, где LLM в основном структурировала текст в ингредиенты, а БЖУ считались из локальных данных. В новом change фиксируется агентский приоритет: пользователь пишет свободный текст, а LLM возвращает итоговую оценку калорий/БЖУ с объяснимостью и уровнем уверенности.

Ограничения:
- MVP-ориентация на скорость логирования и UX-простоту.
- Не требуется высокая нутри-точность, но требуется прозрачность решения агента.
- Справочник не должен быть обязательным звеном в основном потоке.
- Текущий стек: SwiftUI + SwiftData/SQLite + OpenAI/DeepSeek-compatible API.

Ключевые стейкхолдеры:
- Пользователь (быстрое логирование, понятные подсказки по «дорогим» ингредиентам).
- Продукт (retention, частота логирования, стабильность потока).
- Разработка (дебагопригодность через версионирование и raw-ответы).

## Goals / Non-Goals

**Goals:**
- Внедрить agent-first поток логирования: free-text → LLM оценка.
- Зафиксировать строгий JSON-контракт ответа LLM для устойчивого парсинга.
- Поддержать два режима оценки:
  - `ingredient_breakdown` для явно составных текстов;
  - `composite_item` для готовых/брендовых позиций без принудительной декомпозиции.
- Добавить explainability: `confidence`, `impact_score`, `reason`.
- Поддержать lifecycle-статусы (`pending-estimation`, `structured`, `parse_failed`) без retry.
- Сохранять raw LLM JSON для отладки с retention 90 дней.
- Сохранять ручные правки как память и прозрачно показывать применение памяти.
- Версионировать оценку (`modelId`, `promptVersion`, `estimationSchemaVersion`).

**Non-Goals:**
- Построение или поддержка глобального справочника продуктов.
- Достижение медицинской/клинической точности макрооценки.
- Полноценный MCP-memory слой (выносится в следующий change).
- Автоматическая ретро-миграция исторических оценок (приложение новое, legacy не учитывается).

## Decisions

### 1. LLM-first контракт как единственный источник расчёта в happy-path
**Решение:** в основном UX-потоке расчёт калорий/БЖУ приходит из LLM, без обязательного `FoodItem` lookup.

**Почему:** минимизирует трение, убирает dependency на полноту справочника и ускоряет цикл логирования.

**Альтернатива:** гибрид “LLM + справочник как источник истины”.
Отклонено в этом change из-за избыточной операционки и задержки MVP.

### 2. Строгий JSON-контракт вместо свободного текста
**Решение:** LLM обязана вернуть валидный JSON фиксированной структуры.

Минимальный контракт ответа:
- `mode`: `ingredient_breakdown | composite_item`
- `totals`: `calories`, `proteins`, `fats`, `carbs`
- `confidence`: `low | medium | high`
- `items[]`:
  - `name`
  - `estimatedCalories`, `estimatedProteins`, `estimatedFats`, `estimatedCarbs`
  - `impact_score` (0..1)
  - `reason` (краткое объяснение)
  - `high_calorie_flag` (bool)
- `assumptions[]` (ключевые допущения)
- `modelId`, `promptVersion`, `estimationSchemaVersion`

**Почему:** предсказуемое сохранение и UI-рендер без хрупкого парсинга.

**Альтернатива:** markdown/natural-language ответ.
Отклонено из-за parse-fail и нестабильного UX.

Правила валидации контракта:
- `items[]` обязателен всегда; для `composite_item` должен содержать минимум 1 элемент.
- `parse_failed` выставляется и при невалидном JSON, и при валидном JSON с пропущенными обязательными полями.
- Для `ingredient_breakdown` сумма макроэлементов по `items` должна совпадать с `totals` с допуском ±5%.
- `impact_score` валидируется и принудительно ограничивается диапазоном [0..1] на клиенте.
- Explainability обязательна на уровне item (`impact_score`, `reason`) даже в `composite_item`.

### 3. Fail-fast без retry
**Решение:** при невалидном JSON или отсутствующих обязательных полях выставлять `parse_failed`, сохранять raw лог и давать ручную переоценку.

**Почему:** предсказуемый контроль пользователем и минимальная сложность сети/оркестрации.

**Альтернатива:** автоматический retry с более жёстким prompt.
Отклонено по решению продукта (лишняя задержка и усложнение).

### 4. Dual-mode логика оценки
**Решение:**
- если пользователь явно перечисляет состав (“с”, “и”, список компонентов) — `ingredient_breakdown`;
- если это готовый продукт/брендовая позиция — `composite_item` без насильной декомпозиции.

**Почему:** сохраняется прозрачность там, где она полезна, и не создаётся ложная точность для готовых позиций типа Big Tasty.

**Альтернатива:** всегда раскладывать на ингредиенты.
Отклонено, так как часто некорректно для составных брендовых блюд.

### 5. Explainability как обязательный слой
**Решение:** сохранять и показывать `confidence`, `impact_score`, `reason`, с цветовой подсветкой high-impact элементов.

**Почему:** продукту нужна прозрачность решения агента и понятность “почему подсветилось”.

**Альтернатива:** только финальные цифры без объяснения.
Отклонено из-за плохой интерпретируемости и недоверия к агенту.

### 6. Хранение и retention
**Решение:** хранить raw JSON 90 дней, затем очищать raw payload, оставляя нормализованные поля в БД.

**Почему:** баланс между отладкой и объёмом данных на устройстве.

**Альтернатива:** не хранить raw вовсе или хранить бессрочно.
Отклонено: первое ухудшает дебаг, второе увеличивает storage без пользы.

### 7. Память пользовательских правок
**Решение:** ручные правки сохраняются в локальную память маппингов по нормализованному fingerprint; при последующем применении в UI показывается факт использования памяти.

**Почему:** повышает стабильность и персонализацию без скрытых silent override.

**Альтернатива:** одноразовые правки без памяти.
Отклонено: не улучшает поведение агента со временем.

Применение памяти:
- память работает как prior (подсказка), а не как жёсткий override;
- итоговая оценка остаётся результатом текущей обработки агента.

### 8. Non-blocking UX для низкой уверенности
**Решение:** при `confidence = low` лог не блокируется, событие сохраняется, UI показывает предупреждение и быстрые действия.

**Почему:** ключевая цель change — скорость логирования и снижение трения.

**Альтернатива:** блокировать сохранение до подтверждения/правки.
Отклонено: ухудшает UX и снижает частоту логирования.

## Risks / Trade-offs

- **[Нестабильность LLM-оценок]** → Митигация: строгий JSON-контракт, confidence/assumptions, версионирование `promptVersion` и `modelId`.
- **[Ложная уверенность пользователя]** → Митигация: явный confidence-индикатор и explainability в UI.
- **[Parse-fail без retry увеличит долю неоценённых логов]** → Митигация: одна кнопка, ведущая в правку текста и затем в повторную отправку; сохранение raw лога без потери данных.
- **[Рост объёма БД из-за raw payload]** → Митигация: retention 90 дней + purge raw.
- **[Снижение покрытия экзотических кейсов после удаления справочника из happy-path]** → Митигация: ручные правки + память, отдельный future change для MCP-memory.

## Migration Plan

1. Обновить контракт `LLMClient` под новый JSON-формат оценки.
2. Добавить поля хранения оценки и explainability в локальные модели (`MealEvent` или выделенную сущность оценки), включая версии и raw payload.
3. Обновить UI-экран логирования и редактирования:
   - убрать structured-ввод из справочника и шаблоны из основного создания лога;
   - добавить отображение confidence/impact/reason;
   - добавить обработку `parse_failed` (единая кнопка: сначала правка текста, затем повторная отправка).
4. Добавить плановый purge raw payload старше 90 дней.
5. Добавить сохранение и применение пользовательской памяти правок с явной индикацией в UI.
6. Прогнать smoke-сценарии: happy-path, parse_failed, composite/breakdown, low-confidence non-blocking, ручная правка и повторное применение памяти.

Rollback:
- вернуть старый контракт LLM и старый UI-поток;
- игнорировать новые поля оценки в чтении;
- отключить purge-job и memory-слой при необходимости.
