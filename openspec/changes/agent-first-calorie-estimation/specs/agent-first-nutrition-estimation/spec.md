## ADDED Requirements

### Requirement: Agent-first оценка питания из свободного текста
Система SHALL принимать свободный текст приёма пищи и возвращать оценку калорий и БЖУ как основной результат без обязательного обращения к локальному справочнику продуктов.

#### Scenario: Успешная оценка из текста
- **WHEN** пользователь вводит текст «овсянка с орехами и медом» и нажимает «Готово»
- **THEN** система сохраняет лог и отображает оценку `calories/proteins/fats/carbs`, рассчитанную LLM

### Requirement: Dual-mode оценка (ingredient_breakdown/composite_item)
Система SHALL поддерживать два режима ответа агента: `ingredient_breakdown` для явно перечисленного состава и `composite_item` для готовых брендовых позиций.

#### Scenario: Breakdown для составного текста
- **WHEN** пользователь вводит «овсянка с грецкими орехами, медом и сливочным маслом»
- **THEN** LLM возвращает режим `ingredient_breakdown`, а система показывает список компонентов с оценками каждого

#### Scenario: Composite для брендовой позиции
- **WHEN** пользователь вводит «Биг Тейсти»
- **THEN** LLM возвращает режим `composite_item`, а система показывает оценку как единую позицию без принудительной декомпозиции

### Requirement: Explainability и подсветка high-impact компонентов
Система SHALL показывать объяснимость результата агента через `confidence`, `impact_score`, `reason` и подсветку высококалорийных компонентов.

#### Scenario: Подсветка high-impact
- **WHEN** LLM возвращает компонент с высоким `impact_score`
- **THEN** система визуально выделяет компонент и показывает краткую причину (`reason`)

### Requirement: Обработка parse_failed без retry
Система SHALL при невалидном ответе LLM или нарушении схемы помечать лог как `parse_failed` без автоматического retry и предоставлять путь к исправлению текста.

#### Scenario: Невалидный ответ агента
- **WHEN** LLM возвращает JSON с пропущенными обязательными полями
- **THEN** система сохраняет статус `parse_failed` и показывает действие, ведущее к правке текста и повторной отправке

### Requirement: Сохранение и применение памяти пользовательских правок
Система SHALL сохранять ручные правки пользователя как память по нормализованному fingerprint и применять их как prior при последующих похожих логах.

#### Scenario: Применение памяти на похожем вводе
- **WHEN** пользователь ранее вручную скорректировал оценку для похожего текста
- **THEN** система применяет память как prior, показывает факт применения в UI и сохраняет итоговую оценку нового запроса

### Requirement: Хранение raw LLM-ответа с retention 90 дней
Система SHALL хранить сырой JSON-ответ LLM для дебага и аудита в течение 90 дней, после чего удалять raw payload, сохраняя нормализованные поля.

#### Scenario: Очистка устаревшего raw payload
- **WHEN** возраст raw ответа превышает 90 дней
- **THEN** система удаляет raw payload и сохраняет структурированные поля оценки
