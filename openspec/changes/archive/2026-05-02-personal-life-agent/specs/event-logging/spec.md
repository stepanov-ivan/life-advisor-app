## ADDED Requirements

### Requirement: Текстовый ввод приёма пищи
Пользователь SHALL иметь возможность залогировать приём пищи через текстовое поле на карточке слота (Завтрак, Обед, Ужин, Перекус).

#### Scenario: Быстрое логирование текстом
- **WHEN** пользователь открывает карточку незаполненного слота и вводит текст «овсянка с ягодами и кофе»
- **THEN** при нажатии «Готово» создаётся `MealEvent` со статусом `raw`, `rawText` содержит введённый текст, карточка становится жёлтой

#### Scenario: Сохранение без текста
- **WHEN** пользователь нажимает «Готово» с пустым полем ввода
- **THEN** система не создаёт событие и подсвечивает поле ввода как обязательное

### Requirement: Структурированный ввод из истории
Пользователь SHALL иметь возможность выбрать прошлый приём пищи из истории для мгновенного структурированного логирования.

#### Scenario: Копирование вчерашнего завтрака
- **WHEN** пользователь открывает карточку слота «Завтрак» и выбирает «Из истории» → вчерашний завтрак
- **THEN** создаётся `MealEvent` со статусом `structured`, ингредиенты и БЖУ скопированы из выбранного события, карточка становится зелёной

#### Scenario: История пуста
- **WHEN** пользователь открывает «Из истории», но история приёмов пищи пуста
- **THEN** отображается сообщение «Нет сохранённых приёмов»

### Requirement: Ввод из справочника продуктов
Пользователь SHALL иметь возможность выбрать продукты из локального справочника `FoodItem` для структурированного логирования.

#### Scenario: Поиск и выбор продукта
- **WHEN** пользователь открывает карточку слота и выбирает «Из справочника», вводит в поиск «овсян»
- **THEN** отображаются совпадения из `FoodItem`, пользователь выбирает продукт и указывает количество → создаётся `MealEvent` со статусом `structured`, один `Ingredient` с данными из справочника

### Requirement: Структурированный ввод из шаблонов
Пользователь SHALL иметь возможность сохранить текущий набор ингредиентов как шаблон и использовать его для будущих логов.

#### Scenario: Сохранение и использование шаблона
- **WHEN** пользователь на зелёной карточке нажимает «Сохранить как шаблон», даёт название «Мой завтрак»
- **THEN** шаблон сохраняется, и при следующем открытии карточки в разделе «Шаблоны» доступен «Мой завтрак» — один тап создаёт `structured` событие с теми же ингредиентами

### Requirement: Отображение статуса карточки
Система SHALL отображать карточку слота в одном из трёх цветовых состояний в зависимости от статуса `MealEvent`.

#### Scenario: Пустой слот
- **WHEN** приём пищи не залогирован и не помечен как skipped
- **THEN** карточка отображается серым цветом с названием окна

#### Scenario: Сырой ввод
- **WHEN** `MealEvent` имеет статус `raw`
- **THEN** карточка отображается жёлтым цветом, показывает `rawText` и индикатор «Ожидает обработки»

#### Scenario: Структурированные данные
- **WHEN** `MealEvent` имеет статус `structured`
- **THEN** карточка отображается зелёным цветом, показывает сумму калорий, БЖУ и список блюд

#### Scenario: Осознанный пропуск
- **WHEN** `MealEvent` имеет статус `skipped`
- **THEN** карточка отображается серым цветом с иконкой перечёркивания

### Requirement: Редактирование приёма пищи
Пользователь SHALL иметь возможность редактировать залогированный приём пищи: изменить текст, добавить/удалить ингредиенты вручную.

#### Scenario: Редактирование сырого текста
- **WHEN** пользователь тапает по жёлтой карточке и меняет текст «овсянка с ягодами» на «овсянка с бананом»
- **THEN** `rawText` обновляется, статус остаётся `raw`, при необходимости можно запустить структуризацию заново

#### Scenario: Редактирование структурированных ингредиентов
- **WHEN** пользователь тапает по зелёной карточке, меняет количество овсянки с 100г на 150г
- **THEN** `Ingredient.amount` обновляется, агрегированные БЖУ `MealEvent` пересчитываются автоматически

#### Scenario: Добавление ингредиента вручную
- **WHEN** пользователь на зелёной карточке нажимает «Добавить ингредиент», выбирает «банан» из справочника, 100г
- **THEN** создаётся новый `Ingredient`, БЖУ пересчитываются

### Requirement: Удаление приёма пищи
Пользователь SHALL иметь возможность удалить залогированный приём пищи.

#### Scenario: Удаление свайпом
- **WHEN** пользователь делает свайп влево на карточке приёма пищи и нажимает «Удалить»
- **THEN** `MealEvent` и все связанные `Ingredient` удаляются, карточка возвращается в серое состояние
