## Why

Для MVP критична скорость логирования и низкий порог ввода: пользователь должен писать свободным текстом и сразу получать оценку калорий/БЖУ без зависимости от большого справочника. Текущий подход со structured-вводом из справочника и шаблонами усложняет UX и замедляет цикл «ввод → результат».

## What Changes

- Перевести основной поток логирования в agent-first: free-text ввод и оценка калорий/БЖУ через LLM.
- Удалить из основного UX structured-ввод «Из справочника» и использование шаблонов в форме создания лога.
- Ввести строгий JSON-контракт ответа LLM для стабильного парсинга и отображения.
- Ввести статусный fallback без retry: при ошибке парсинга сохранять событие как `parse_failed` с возможностью переоценки.
- Добавить confidence-оценку от LLM и цветовую подсветку в UI.
- Ввести dual-mode оценку:
  - `ingredient_breakdown` для явно составных текстов (показывать ингредиенты по отдельности);
  - `composite_item` для готовых/брендовых позиций (Big Tasty/Big Mac и т.п.) без принудительной декомпозиции.
- Добавить подсветку высококалорийных компонентов на базе LLM-решения, но с обязательными полями `impact_score` и `reason`.
- Сохранять пользовательские ручные правки как память для последующих похожих логов, с прозрачным указанием, когда память была применена.
- Сохранять raw JSON-ответ LLM для отладки и аудита с retention-политикой: хранить 90 дней, затем удалять raw payload, оставляя нормализованные поля.
- Версионировать результат оценки (`modelId`, `promptVersion`, `estimationSchemaVersion`) для дебага и сопоставимости.

## Capabilities

### New Capabilities
- `agent-first-nutrition-estimation`: LLM-first оценка калорий/БЖУ из свободного текста с confidence, dual-mode (composite/breakdown), explainability и lifecycle-статусами (`structured`, `parse_failed`, `pending-estimation`).

### Modified Capabilities
- `event-logging`: Изменение требований к потоку логирования — основной вход только через текст и агентскую оценку; удаление structured-ввода из справочника и шаблонов из основного потока; добавление UI для confidence/impact/reason.
- `llm-agent`: Изменение требований к контракту ответа — строгий JSON с макрооценкой, confidence, mode, компонентами, impact-объяснениями, версионированием и сохранением raw ответа.
- `food-reference`: Изменение требований — справочник перестаёт быть обязательным источником расчёта в основном UX-потоке; возможен fallback/обогащение в будущих изменениях.
- `local-storage`: Изменение требований к хранению оценки: raw JSON, explainability-поля, версии оценки, memory-флаги и retention raw данных 90 дней.

## Impact

- Затронутые экраны: `DashboardView`, `LogSheetView`, `MealEventEditorView`.
- Затронута LLM-интеграция: `LLMClient` (новый строгий контракт и поля объяснимости).
- Затронуто хранение: модель `MealEvent` и/или связанные сущности оценки для confidence, impact, reason, raw payload и версий.
- Затронута бизнес-логика постобработки: статусы `parse_failed`/`pending-estimation`, переоценка по кнопке, применение и отображение пользовательской памяти.
- Снижение продуктовой сложности на MVP-этапе за счёт удаления обязательного справочника из happy-path.
