## Why

Текущий блок `Состав блюда` перегружен лишним действием (`Сбросить к автопересчёту`) и неоднозначностью единиц измерения (`g/ml/pcs`). Это усложняет ввод, делает автопересчёт непредсказуемым и приводит к ошибкам интерпретации порций. Для упрощения UX и стабилизации расчётов нужно перейти на единый формат: только граммы.

## What Changes

- Удалить из UI секции `Состав блюда` кнопку `Сбросить к автопересчёту`.
- Зафиксировать single-path поведение: изменение граммовки всегда запускает автопересчёт и перезатирает ручные макросы для этой позиции.
- Перейти на grams-only контракт LLM:
  - `item.name` — чистое имя продукта/компонента,
  - `item.grams` — обязательное числовое значение (> 0).
- Удалить из контракта и UI понятие `unit`.
- Обновить UI таблицы:
  - убрать колонку `ед`,
  - переименовать `кол-во` в `г`.
- Сохранить поддержку дробных значений:
  - `grams` — дробные разрешены,
  - отображение `Ккал/Б/Ж/У` — до 1 знака после запятой.
- Удалить из модели поля, связанные с unit, и переименовать `quantity` в `grams`.
- Выполнить destructive schema change без data migration (сброс несовместимых данных допустим).
- Сохранить строгий режим ошибок: при нарушении контракта LLM оценка падает целиком в `parseFailed`.

### Modified Capabilities
- `event-logging`: изменение требований редактора `Состав блюда` (удаление кнопки ручного сброса, grams-only ввод, авто-пересчёт как единственный сценарий).
- `llm-agent`: изменение контракта оценки структуры блюда (переход с `quantity + unit` на `grams`).

## Impact

- UI: `DashboardView` (редактор приёма пищи, таблица `Состав блюда`).
- Domain model: `EstimateItem` (переименование поля и удаление unit-полей).
- LLM integration: контракт/валидация `estimateMeal` и связанные тесты.
- Хранилище: несовместимое изменение схемы с destructive reset (без миграции данных).
- Тесты: обновление unit/contract тестов под новый формат `grams`.
