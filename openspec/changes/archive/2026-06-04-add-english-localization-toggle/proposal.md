## Why

Сейчас приложение ориентировано в первую очередь на русскоязычный интерфейс, из-за чего его сложнее показывать англоязычным пользователям, контрибьюторам и внешним ревьюерам. Частичная локализация здесь не решает задачу: для продукта важны не только переведённые экраны, но и консистентный язык уведомлений, объяснений правил, форматирования, accessibility-текстов и LLM-ответов. Полноценная локализация с явным переключением языка в настройках нужна сейчас, чтобы сделать продукт понятнее, повысить качество публичной витрины и подготовить основу для дальнейшего роста OSS-проекта.

## What Changes

- Добавить полноценную локализацию продукта на русский и английский языки с английским как базовым fallback-языком.
- Добавить выбор языка в настройках приложения в формате `System / English / Русский` с сохранением пользовательского override между запусками.
- Определять язык приложения по системной локали по умолчанию, но разрешать пользователю переопределять его вручную.
- Применять изменение языка сразу, без перезапуска приложения.
- Вынести все user-facing hardcoded строки из интерфейса в стандартную Apple localization infrastructure.
- Локализовать обычный UI, accessibility-тексты, alerts, placeholders, tab labels, notifications, ошибки, rule presentation, prompt templates и форматирование дат/чисел/единиц.
- Сохранить rules как язык-независимую структуру и поставлять локализацию titles/descriptions/reason labels отдельным presentation layer поверх stable ids.
- Обеспечить, чтобы уведомления, user-facing ошибки и новый LLM-контент использовали выбранный язык приложения.
- Добавить автоматические проверки полноты локализаций и тесты централизованного language-selection/fallback поведения.
- Обновить документацию для разработчиков и контрибьюторов по новой архитектуре локализации.

## Capabilities

### New Capabilities
- `ui-localization`: инфраструктура локализации приложения, поддержка языков `ru` и `en`, runtime-переключение языка в настройках, English fallback и централизованное управление effective app language.

### Modified Capabilities
- `app-setup`: запуск приложения и экран настроек должны использовать system language по умолчанию, поддерживать пользовательский override и применять его ко всему интерфейсу без перезапуска.
- `nutrition-rules`: пользовательские названия, описания, категории, причины нарушений и отображаемые пороги правил должны локализоваться отдельно от rule logic и не влиять на stable rule ids.
- `rule-violation-tracing`: пользовательские объяснения нарушений и предупреждений должны формироваться на активном языке приложения и не зависеть от локализуемого display text.
- `russian-violation-reasons`: существующие русскоязычные причины нарушений должны быть преобразованы в двуязычное пользовательское представление с общими language-agnostic identifiers.

## Impact

- Затронутые области кода: SwiftUI views, settings flow, onboarding, analytics, dashboard, rules screens, accessibility labels/hints, notification texts, сервисные user-facing строки, prompt templates и formatting logic.
- Потребуется единый `AppLanguageManager`/`LocalizationManager`, который хранит пользовательский override, вычисляет effective language и пробрасывает locale в UI и non-UI код.
- Появятся отдельные localization tables/resources для обычного UI, rules presentation и prompt templates.
- Потребуются изменения в локальном хранении пользовательских настроек языка через `UserDefaults`/`@AppStorage`.
- Понадобятся тесты на полноту локализаций и на централизованную логику выбора языка/fallback.
- Нужно будет обновить developer-facing documentation о том, как добавлять новые локализуемые строки и не завязывать поведение на display text.

## Non-Goals

- Не переводить уже сохранённый пользовательский или исторический контент.
- Не делать data migration ради локализации.
- Не строить отдельный кастомный localization framework вместо стандартного Apple stack.
- Не проектировать систему под произвольное количество языков сверх текущих `ru` и `en`.
- Не добавлять публичный переключатель языка до готовности реальной локализации продукта.
- Не делать отдельную пользовательскую настройку языка только для LLM.
- Не требовать полных end-to-end тестов качества всех LLM-ответов в рамках этого change.
- Не локализовать debug-only и developer-only строки, если их не видит пользователь.
