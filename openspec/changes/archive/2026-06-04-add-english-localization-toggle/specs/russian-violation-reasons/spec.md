## MODIFIED Requirements

### Requirement: Генерация локализованных описаний нарушений
Система SHALL предоставлять локализованное пользовательское описание нарушения на основе `RuleViolation.reasonCode`, stable rule ids и presentation model. Описание SHALL поддерживать как минимум `ru` и `en` и SHALL использовать active app language вместо жёстко зашитого русского текста.

#### Scenario: Превышение верхней границы на русском языке
- **WHEN** effective language приложения русский, нарушение имеет `reasonCode = "exceeds_upper"`, а правило `who_free_sugar` превышает верхнюю границу
- **THEN** пользователь видит локализованное русское описание нарушения, построенное через presentation layer

#### Scenario: Превышение верхней границы на английском языке
- **WHEN** effective language приложения английский, нарушение имеет `reasonCode = "exceeds_upper"`, а правило `who_free_sugar` превышает верхнюю границу
- **THEN** пользователь видит локализованное английское описание того же нарушения, построенное через presentation layer

#### Scenario: Отсутствие категории на английском языке
- **WHEN** effective language приложения английский и нарушение имеет `reasonCode = "category_missing"` для правила `who_whole_grain`
- **THEN** пользователь видит английское описание отсутствующей категории, локализованное по stable category id

### Requirement: Использование локализованных описаний в UI
Система SHALL использовать локализованные описания нарушений во всех местах, где пользователю показывается explanation text нарушения, вместо показа технических `reasonCode` или жёстко зашитых русских шаблонов.

#### Scenario: Карточка приёма на английском языке
- **WHEN** `MealSlotCard` показывает нарушение правила при активном английском языке
- **THEN** отображается английское локализованное описание нарушения вместо `reasonCode`

#### Scenario: Детальный экран правила на русском языке
- **WHEN** `RuleDetailView` показывает список нарушений при активном русском языке
- **THEN** explanation text каждого нарушения отображается как русское локализованное описание вместо технического `reasonCode`

### Requirement: Сохранение reasonCode в модели
Система SHALL сохранять оригинальный `reasonCode` в модели `RuleViolation` без изменений. Локализованное описание SHALL генерироваться на лету и использоваться только в presentation/UI-слое.

#### Scenario: Обратная совместимость модели нарушений
- **WHEN** существующие тесты или логика проверяют `violation.reasonCode`
- **THEN** значение `reasonCode` остаётся прежним и не зависит от выбранного языка приложения
