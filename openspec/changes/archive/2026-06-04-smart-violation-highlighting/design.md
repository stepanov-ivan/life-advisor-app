## Context

Текущая система подсветки нарушений в `MealSlotCard` использует статический порог 30%: любой продукт с вкладом >30% в дневной нутриент получает цветной индикатор, независимо от типа нарушения. Это создаёт шум: кефир подсвечивается за отсутствие клетчатки, курица — за сахар, хотя сахара в ней нет. Пользователь видит 5+ нерелевантных нарушений на одном продукте.

Экран `RulesListView` показывает зоновые индикаторы для каждого правила, хотя его основная функция — настройка вкл/выкл.

Текстовки нарушений используют формат «Превышение: значение при норме порог» без названия правила.

## Goals / Non-Goals

**Goals:**
- Сделать подсветку продуктов релевантной: только превышения (exceeds_upper, approaching_upper) и нежелательные категории (unwanted_category_present)
- Убрать нерелевантные подсветки: category_missing, below_lower, approaching_lower
- Поднять порог вклада с 30% до 33%
- Добавить название правила в текстовку нарушения на карточке приёма
- Превратить экран правил в чисто настроечный

**Non-Goals:**
- Не менять `DayAnalyticsPanel`
- Не менять `RuleDetailView`
- Не менять модель `RuleViolation`
- Не добавлять ИИ/LLM
- Не менять логику вычисления самих нарушений (evaluateRange, evaluatePresence остаются без изменений)

## Decisions

### 1. Где применять фильтр по reasonCode

**Решение:** Нарушения с `below_lower`, `approaching_lower` и `category_missing` не должны попадать в карточки приёма — это нарушения дня, а не конкретного приёма или продукта. Фильтрация на двух уровнях:

- `RuleEngine.contributionForPresenceRule` — возвращает пустой массив для `inverse: false` (category_missing). contributionJSON = nil → `violationsForMeal` не матчит приём → нарушение не показывается в карточке
- `MealSlotCard` — фильтрует входящий массив `violations`: пропускает нарушения с `reasonCode = "below_lower"` или `"approaching_lower"` перед отрисовкой. Эти нарушения не показываются ни в списке нарушений карточки, ни в подсветке продуктов

**Альтернатива:** Фильтровать в `violationsForMeal`. Отвергнуто — метод используется и для других целей, лучше фильтровать на уровне UI.

**Почему не в contributionForRangeRule:** На момент вызова `contributionMap` reasonCode ещё не вычислен (он определяется в `generateViolations` после `evaluateRule`). Contribution и reasonCode разделены во времени.

### 2. Формат violationDescription

**Решение:** Изменить формат с «Превышение: 14,2% при норме ≤10%» на «Добавленный сахар: 14,2% при норме ≤10%».

Метод `violationDescription(for:)` в `RuleEngine` уже имеет доступ к `rule.title` через `rules.first(where:)`. Заменяем префикс reasonCode-текста на название правила с двоеточием.

**Альтернатива:** Разные форматы для RuleDetailView и MealSlotCard. Отвергнуто — пользователь подтвердил, что унифицированный формат норм.

### 3. Экран правил — что удалить

**Решение:** Удалить из `RulesListView`:
- `summary`-секцию (строка «N из M» + круговой индикатор)
- `zoneIndicator` (цветной кружок)
- `zoneLabel` (текст «В норме»/«Нарушено»)
- Вызов `engine.evaluateToday(rule:)` для каждой строки

Оставить:
- Секции по категориям
- Название и описание правила
- Toggle вкл/выкл

### 4. Порог вклада 33%

**Решение:** Изменить константу с 0.30 на 0.33 в `contributionForRangeRule` (строка `$0.percent > 30` → `$0.percent > 33`).

Поведение: если ни один продукт не превышает 33%, contributionJSON пуст → продукты не подсвечиваются → нарушение только на уровне дня.

### 5. contributionForPresenceRule — исключение category_missing

**Решение:** Для `inverse: false` (правила типа «должна быть категория X») возвращать пустой массив. Эти нарушения — отсутствие категории в дне, а не вина конкретного продукта.

Для `inverse: true` (нежелательная категория) — оставить как есть. Продукт с нежелательной категорией — прямая причина нарушения.

Код:
```swift
private func contributionForPresenceRule(_ rule: RuleDefinition, date: Date, context: ModelContext) -> [ContributionEntry] {
    // category_missing: не подсвечиваем продукты — это нарушение дня, а не продукта
    let inverse = rule.params.inverse ?? false
    guard inverse else { return [] }
    // ... остальная логика для unwanted_category_present
}
```

### 6. Карточка приёма без релевантных нарушений

**Решение:** Если после фильтрации в карточке не осталось релевантных нарушений (все below_lower, approaching_lower или category_missing) — не показываем ничего. Ни зелёную плашку «Все правила соблюдены», ни список нарушений. Секция нарушений просто отсутствует.

### 7. contributionForRangeRule для правил с нижней границей

**Решение:** Поля `fiberGrams`, `fruitVegGrams` — правила с нижней границей (≥25 г). Их contribution вычисляется, но в UI не показывается (below_lower фильтруется). Вычисление остаётся как есть — экономия от раннего выхода незначительна, а усложнение кода неоправдано.

## Risks / Trade-offs

- **Ложно-положительные срабатывания для сахара:** Курица может получить contribution >33% по сахару, если в дне очень мало сахара (например, 3 г из курицы из 5 г total = 60%). Это решается на уровне данных — estimatedSugar для курицы должен быть ~0. Если модель EstimateItem даёт корректные оценки, проблема исчезает.
- **Порог 33% может быть слишком высоким:** Если у пользователя 2 продукта с вкладом 32% каждый, ни один не подсвечивается, хотя вместе они дают 64% нарушения. Это осознанный trade-off — лучше пропустить, чем показать шум.
- **Удаление зоновых индикаторов с экрана правил:** Пользователь теряет быстрый обзор статуса всех правил на одном экране. Эта информация остаётся доступной в DayAnalyticsPanel на дашборде.
