## Context

Текущий `DashboardView` ориентирован только на сегодняшний день и не поддерживает удобный просмотр истории по датам. Интерфейс карточек и редактора содержит технические элементы, которые полезны для отладки, но перегружают пользовательский UX. В результате пользователь хуже понимает, что является основной структурой блюда и как интерпретировать блок советов.

Ограничения и вводные:
- стек: SwiftUI + SwiftData;
- структура блюда остаётся источником истины;
- LLM-парсинг структуры должен оставаться явным действием пользователя (по кнопке), без автозапросов на каждый ввод;
- даты в будущем должны быть недоступны для выбора, т.к. флоу ориентирован на разбор уже состоявшихся приёмов пищи и корректировку поведения;
- границы дня считаются в локальной таймзоне устройства (`Calendar.current`).

Стейкхолдеры:
- конечный пользователь (понятный и быстрый UX без технического шума);
- разработка (предсказуемая модель синхронизации текст/структура и тестируемость).

## Goals / Non-Goals

**Goals:**
- Добавить навигацию по датам на дашборде: week-strip (7 дней) + календарь для точечного выбора.
- Ограничить выбор дат до `today` включительно.
- Скрыть технические метки (`Уверенность`, `Из памяти`) из основной карточки приёма пищи.
- Упростить и переименовать блок структуры в `Состав блюда`.
- Добавить табличный формат структуры: `кол-во | ед | Ккал | Б | Ж | У`.
- Поддержать единицы количества `g / ml / pcs` с дефолтом `g`.
- Разрешить дробные значения количества.
- Переименовать блок `Почему этот совет` в `Совет`, убрать технические explainability-поля и кнопку `Показать всё`.
- Для событий из памяти показывать совет только при наличии осмысленного текста рекомендации.
- При редактировании текста автоматически выставлять устаревание структуры, но не запускать LLM до явного нажатия кнопки.

**Non-Goals:**
- Изменение алгоритма расчёта БЖУ/ккал в LLM-контракте.
- Добавление новых источников данных или справочников.
- Автоматическая конвертация единиц `g/ml/pcs` в рамках этого change.
- Рефакторинг всех экранов приложения за пределами дашборда/редактора/блока совета.

## Decisions

### 1) Навигация по датам: календарная неделя + календарь
Решение:
- Добавить сверху дашборда compact week-strip на 7 дней (календарная неделя `Пн...Вс`) и кнопку календаря.
- При выборе даты из календаря перестраивать week-strip на неделю выбранной даты.
- Открывать календарь на месяце выбранной даты.
- Сохранять последнюю выбранную дату в `@AppStorage` (day-key `yyyy-MM-dd`) и восстанавливать её при повторном открытии экрана.
- Если сохранённая дата в будущем — fallback на `today`.

Почему:
- week-strip ускоряет сценарии «вчера/сегодня»;
- календарь покрывает быстрый переход к произвольной дате;
- восстановление состояния делает UX непрерывным между сессиями.

Альтернативы:
- встроенный полноразмерный календарь: занимает лишнее место;
- только DatePicker: хуже для быстрых повседневных переключений;
- хранить дату только в `@State`: теряется состояние между перезапусками.

### 2) Ограничение дат: только `<= today`
Решение:
- Будущие даты видимы, но disabled в strip и календаре.
- При попытке выбора будущей даты показывать короткий хинт «Будущие даты недоступны».
- Фильтрация `MealEvent` выполняется по локальному дню устройства через `Calendar.current.startOfDay`/endOfDay.

Почему:
- соответствует продуктовой логике коррекции питания по факту;
- сохраняет предсказуемые границы дня;
- хинт объясняет ограничение прямо в UI.

Альтернативы:
- разрешить будущие даты: противоречит текущему флоу;
- скрывать будущие даты полностью: хуже объяснимость ограничения.

### 3) Упрощение карточек и блока совета
Решение:
- Удалить из основной карточки технические метки (`Уверенность`, `Из памяти`).
- Переименовать блок `Почему этот совет` в `Совет`.
- Убрать служебные факторы (`источник структуры`, `уверенность модели`, `режим оценки`) и кнопку `Показать всё` из пользовательского UI.
- Для memory-событий показывать совет только если текст проходит полезность: `trim != empty`, длина >= 40 символов, не служебный шаблон.

Почему:
- снижает когнитивную нагрузку и делает UI «человеческим», а не диагностическим.

Альтернативы:
- оставить техполя с раскрытием: перегружает интерфейс;
- вводить новый LLM-флаг `isActionable` сразу: потребует отдельного изменения контракта.

### 4) Секция структуры: `Состав блюда` и табличные столбцы
Решение:
- Переименовать секцию ручной правки в `Состав блюда`.
- Ввести явные заголовки столбцов: `кол-во | ед | Ккал | Б | Ж | У`.
- Добавить количество и единицу в модель редактируемой позиции.
- Единицы фиксированные: `g`, `ml`, `pcs`; дефолт `g`.
- Разрешить дробные значения количества.
- При смене единицы менять только label единицы, без автоконверсии значений в этом change.

Почему:
- структура становится читаемой и редактируемой как таблица;
- фиксированные единицы упрощают валидацию и тестирование;
- отсутствие автоконверсии избегает ложной точности без данных плотности.

Альтернативы:
- свободный текст единицы: выше риск мусорных значений;
- автоконверсия сразу: риск ошибок и скрытой логики.

### 5) Синхронизация текста и структуры
Решение:
- При изменении текста после подстановки/существующей структуры помечать структуру как устаревшую (user-facing текст без термина `out_of_sync`, например: «Текст изменён. Структура может быть неактуальна.»).
- Не изменять структуру автоматически и не вызывать LLM при вводе.
- Пересборка структуры через LLM выполняется только по явной кнопке `Пересчитать по тексту`.
- Если LLM-ответ невалиден/ошибка сети — сохранять текущую структуру и показывать ошибку.
- Если пользователь очистил/удалил все позиции структуры — запрашивать подтверждение и переводить событие в `pending-estimation`.

Почему:
- защищает от потери данных;
- устраняет гонки запросов и лишние LLM-вызовы;
- делает действия пользователя явными и обратимыми.

Альтернативы:
- auto-LLM с debounce: дороже и менее предсказуемо;
- автоматическая очистка структуры на каждый ввод: риск случайной потери правок.

## Risks / Trade-offs

- [Пользователь не видит происхождение данных на карточке] → Служебные детали остаются в логах/debug, а в UI сохраняется только полезная информация.
- [Локальное хранение последней даты может устареть] → Проверка на старте и fallback на `today`.
- [Сужение единиц до g/ml/pcs может не покрыть редкие кейсы] → Для текущего MVP это осознанный компромисс; расширение enum отдельным change.
- [Отсутствие автоконверсии единиц] → Явно обозначено в UX и документации как текущая граница функциональности.
- [Heuristic полезности совета может отфильтровать полезный короткий текст] → Порог длины и правила фильтрации вынести в настраиваемую константу с последующей калибровкой.

## Migration Plan

1. Добавить состояние выбранной даты в дашборд, фильтрацию `MealEvent` по выбранному дню и хранение даты в `AppStorage`.
2. Добавить week-strip (Пн...Вс) и calendar picker с блокировкой будущих дат и хинтом.
3. Упростить карточку приёма пищи: убрать технические метки.
4. Переименовать и переработать секцию структуры в `Состав блюда` с табличными столбцами и количеством/единицей.
5. Добавить/обновить кнопку `Пересчитать по тексту` и дружелюбную индикацию устаревшей структуры.
6. Обновить блок `Совет`: убрать техфакторы и `Показать всё`, применять фильтр полезности текста.
7. Зафиксировать обработку ошибок LLM: оставлять текущую структуру, показывать ошибку.
8. Добавить подтверждение при очистке всех позиций структуры и перевод в `pending-estimation`.
9. Обновить тесты:
- unit/integration: фильтрация по дате, out-of-sync поток, полезность совета;
- UI: выбор даты в календаре меняет список событий; будущая дата недоступна и показывает хинт.

Rollback:
- Вернуть предыдущую версию `DashboardView`/редактора/блока совета из git и отключить новые UI-поля в текущем change.
