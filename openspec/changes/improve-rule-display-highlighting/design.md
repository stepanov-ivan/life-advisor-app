## Context

Сейчас `DayAnalyticsPanel` показывает для каждого правила только цветной кружок и зону («В норме», «Внимание», «Нарушено»). Пользователь не видит ни фактического значения метрики, ни порога. `MealSlotCard` показывает `violation.reasonCode` — технические английские строки (`exceeds_upper`, `below_lower`). Данные о вкладе продуктов уже вычисляются и хранятся в `contributionJSON`, но не визуализируются в карточке приёма.

Стек: SwiftUI, SwiftData, RuleEngine как @Observable-сервис. Все 16 правил загружаются из `nutrition_rules.json`. `RuleViolation` — SwiftData-модель с полями `ruleId`, `date`, `zone`, `magnitude`, `reasonCode`, `contributionJSON`.

## Goals / Non-Goals

**Goals:**
- Показывать в `DayAnalyticsPanel` числовое значение метрики и порог правила вместо зонового текста
- Заменить английские `reasonCode` на человекочитаемые русские описания в `MealSlotCard` и `RuleDetailView`
- Подсвечивать продукты в карточке приёма, чей вклад в нарушенное правило >30%, с раскрытием через `DisclosureGroup`
- Добавить сводную строку «N из M правил соблюдаются» над группами правил в панели
- Сохранить обратную совместимость — модель `RuleViolation` не меняется

**Non-Goals:**
- Изменение логики вычисления правил
- Изменение JSON-конфигурации правил (nutrition_rules.json остаётся без изменений)
- Добавление новых полей в SwiftData-модели
- Изменение цветовой схемы зон

## Decisions

### 1. Визуальный макет DayAnalyticsPanel

Строка правила меняется с:
```
[●] Белок в норме                    В норме
```
на:
```
[●] Белок в норме               14,2% из 10–15%
```

- **Число справа** заменяет зоновый текст («В норме» / «Нарушено»)
- **Цвет числа** соответствует зоне: зелёный (normal), жёлтый (warning), красный (violation), серый (no data)
- **Цветной кружок слева** сохраняется — дублирует зону для быстрого считывания
- **Сводная строка** «N из M правил соблюдаются» добавляется над группами, цвет — по худшей зоне среди всех правил

### 2. Форматирование значений метрик — маппинг field → единица измерения

Создаётся вспомогательная структура в `RuleEngine`, которая по `field` правила определяет единицу измерения и формат числа:

| field | unit | multiply | decimals |
|---|---|---|---|
| `*Percent` (protein, fat, carbs, saturatedFat, transFat, sugar, energyBalance, pufa) | `%` | ×100 | 1 |
| `sodiumMg` | ` мг` | ×1 | 0 |
| `fiberGrams`, `fruitVegGrams`, `redMeatGrams` | ` г` | ×1 | 0 |

**presence-правила** — без числового формата, показывают зоновый текст («В норме» / «Нарушено»).

**countSkipped** — показывают количество пропусков: «3 пропуска из ≤3».

Метод `RuleEngine.formattedMetric(for:date:) -> RuleMetricDisplay` вычисляет текущее значение правила, применяет формат и возвращает структуру с полями `valueFormatted`, `thresholdFormatted`, `unit`, `zone`.

**Почему не в View:** форматирование зависит от знания `field` и `params` правила, это доменная логика. View остаётся тонким — только отображает готовые строки.

### 3. Русские описания нарушений — генерация на лету

Метод `RuleEngine.violationDescription(for:) -> String` генерирует описание нарушения по `RuleViolation`, находя `RuleDefinition` по `ruleId` и форматируя в зависимости от `reasonCode`:

| reasonCode | Шаблон |
|---|---|
| `exceeds_upper` | «Превышение: {value}{unit} при норме ≤{upper}{unit}» |
| `below_lower` | «Недостаточно: {value}{unit} при норме ≥{lower}{unit}» |
| `approaching_upper` | «Близко к пределу: {value}{unit} при норме ≤{upper}{unit}» |
| `approaching_lower` | «Близко к минимуму: {value}{unit} при норме ≥{lower}{unit}» |
| `category_missing` | «Не хватает продуктов категории "{categoryTitle}"» |
| `unwanted_category_present` | «Присутствуют нежелательные продукты: {categoryTitle}» |
| `excessive_skips` | «Много пропусков: {count} за неделю» |
| `some_skips` | «Есть пропуски приёмов: {count} за неделю» |

Метод используется и в `MealSlotCard`, и в `RuleDetailView`.

**Альтернатива (не выбрана):** хранение готового описания в `RuleViolation` при создании. Отвергнуто — требует миграции модели и дублирует данные, которые можно вычислить.

### 4. Формат нарушений в карточке приёма (MealSlotCard)

Вместо английского `reasonCode` карточка показывает:
- **Название правила** (русское, из `RuleDefinition.title`)
- **Вклад приёма** в нарушение: процент из `contributionJSON`

Формат: `«Свободный сахар — 45% дневного»`

Процент — максимальный `percent` среди продуктов этого приёма в `contributionJSON` нарушения. Максимум 3 нарушения в карточке.

### 5. Подсветка продукта — три уровня вложенности

Три уровня подсветки:
1. **Правила питания** — `DayAnalyticsPanel` (сводка по всем правилам за день)
2. **Приём пищи** — `MealSlotCard` (нарушения, связанные с этим приёмом)
3. **Продукт внутри приёма** — вложенный список нарушений под продуктом

Механика уровня 3:
- Продукт с вкладом >30% в нарушение получает **цветной кружок** (красный/жёлтый по зоне) и **стрелку раскрытия**
- Используется нативный `DisclosureGroup` (SwiftUI)
- При раскрытии — список нарушенных правил с вкладом продукта:
  ```
  Свободный сахар — 45% дневного
  Насыщенные жиры — 32% дневного
  ```
- Один продукт может раскрываться в несколько нарушений, если его вклад >30% в нескольких правилах

Сборка данных в `MealSlotCard`:
1. Парсит `contributionJSON` всех нарушений приёма
2. Строит словарь `[itemID: [(ruleId, ruleTitle, percent, zone)]]`
3. При рендеринге `EstimateItem` проверяет наличие в словаре
4. Если есть — оборачивает в `DisclosureGroup` с цветным кружком

**Почему DisclosureGroup, а не кастом:** нативный компонент, не требует состояния, меньше кода.

### 6. Сохранение reasonCode в модели

Поле `reasonCode` в `RuleViolation` **сохраняется как есть** — оно используется для программной логики (фильтрация, тесты). Русское описание генерируется отдельным методом и используется только в UI. Это не ломает существующие тесты и не требует миграции.

## Risks / Trade-offs

- **Генерация описания на лету требует evaluateRule** — лишний вызов `evaluateRule` для каждого нарушения. При 16 правилах и нескольких нарушениях — пренебрежимо мало (<1ms на локальном SQLite).
- **Формат числовых значений жёстко привязан к field** — при добавлении нового field в JSON нужно обновить `RuleMetricFormat`. Смягчение: маппинг централизован в одном методе.
- **Международные форматы чисел** — форматирование использует локаль устройства. Для русского языка десятичный разделитель — запятая. Это корректно для ЦА.
- **DisclosureGroup внутри ScrollView** — может быть конфликт скролла при раскрытии. Митигация: продуктов-нарушителей обычно 0–2 на приём, раскрытие не создаёт значительного сдвига контента.
