## Context

Текущий поток логирования питания в приложении agent-first опирается на свободный текст и LLM-оценку, но не имеет устойчивого, объяснимого слоя памяти, который уменьшает повторные уточнения и повышает релевантность рекомендаций. При этом уже существует локальное хранилище (SwiftData), структура логов питания и экран редактирования.

Ключевые ограничения:
- память должна быть минимальной и объяснимой, без трекинга навигационного поведения пользователя;
- основной ввод остаётся свободным текстом;
- структура (ккал/БЖУ/граммы) должна быть источником истины;
- подсказки должны работать локально и быстро, без LLM-запроса на каждый символ.

Стейкхолдеры:
- конечный пользователь (быстрый ввод, прозрачные рекомендации, контроль памяти);
- продукт/разработка (предсказуемая архитектура памяти и расширяемость под будущие изменения).

## Goals / Non-Goals

**Goals:**
- Ввести минимальную модель памяти питания с тремя типами сигналов: структурные логи, гипотезы предпочтений/ограничений, data-gap.
- Добавить локальные подсказки ввода из последних 50 структурированных логов (префикс + похожесть).
- Закрепить режим источника истины: структура первична, текст вторичен.
- Ввести состояние `out_of_sync` при расхождении текста и структуры.
- Обеспечить мягкое применение памяти агентом (без блокировок ввода) и объяснимость источника оценки (`memory_suggestion` / `llm`).
- Дать пользователю контроль памяти: точечное удаление и полный сброс.

**Non-Goals:**
- Полноценный рекомендательный движок продуктов/меню.
- Подключение внешних справочников питания или RAG-баз в рамках этого change.
- Онлайн-автодополнение через LLM в real-time.
- Построение сложного поведенческого профиля на кликах/сценариях использования приложения.

## Decisions

### 1) Модель данных памяти
Решение:
- добавить сущности/поля для:
  - `MemorySuggestion` (нормализованный шаблон структуры для быстрого ввода),
  - `MemoryHypothesis` (косвенные выводы о предпочтениях с confidence и lifecycle),
  - `MemoryDataGap` (пробелы данных: например, отсутствующий вес),
  - метаданные источника структуры и синхронности (`source`, `out_of_sync`).

Почему:
- разделение по типам сигналов повышает объяснимость и управляемость.

Альтернативы:
- хранить всё в одном JSON-поле события: проще старт, хуже поддержка/поиск/эволюция схемы.
- хранить только aggregated-факты без гипотез и data-gap: недостаточно для мягких уточнений и прозрачности.

### 2) Источник истины и синхронизация
Решение:
- структура является канонической версией лога;
- текст — быстрый пользовательский ввод;
- при изменении текста после подстановки структуры выставлять `out_of_sync=true`;
- при отправке в `out_of_sync` по умолчанию запускать «Переоценить по тексту»;
- если пользователь очистил текст после подстановки, связанная структура сбрасывается сразу.

Почему:
- исключает тихое расхождение данных и повышает доверие к итоговым значениям.

Альтернативы:
- текст как источник истины: приводит к потере управляемости при ручных правках структуры.
- автоматический silent-resync без флага: непрозрачно для пользователя.

### 3) Подсказки ввода
Решение:
- использовать только локальный индекс последних 50 структурированных логов;
- похожесть определять по нормализованной строке (без LLM-семантики на этапе MVP);
- ранжирование: комбинация частоты и свежести;
- отображать максимум 5 подсказок;
- выбор подсказки подставляет текст и структуру, но не отправляет автоматически.

Почему:
- быстрый отклик, нулевая стоимость инференса, низкий риск деградации UX.

Альтернативы:
- LLM на каждый ввод: дорого, нестабильно по latency.
- только префикс: хуже покрытие опечаток.

### 4) Работа с data-gap и confidence
Решение:
- если вес/объём не указан, лог не блокируется;
- сохраняется `low confidence` + запись `data-gap`;
- показывается быстрый UI для уточнения порции;
- `data-gap` закрывается автоматически после 2 повторных уточнений для похожего блюда.

Почему:
- сохраняет скорость логирования и постепенно улучшает качество данных.

Альтернативы:
- блокировать сохранение без веса: высокий friction.
- молча подставлять значение всегда: низкая прозрачность и риск систематической ошибки.

### 5) Жизненный цикл гипотез
Решение:
- косвенные выводы хранятся как гипотезы;
- подтверждение запрашивается после 2 повторяющихся сигналов;
- UI подтверждения: inline-чип с действиями `Да / Нет / Спросить позже`;
- при `Спросить позже` повторный вопрос не ранее чем через 3 дня;
- после подтверждения хранится мягкое предпочтение;
- перепроверка — только при явном конфликте и с cooldown 7 дней.

Почему:
- снижает ложноположительные выводы и уменьшает раздражающие вопросы.

### 6) Обновление памяти после ручной правки
Решение:
- обновлять шаблон памяти только если изменение калорий <= 30% от исходной структуры;
- если изменение >30%, сохранять как `candidate` без немедленного влияния на основной шаблон.

Почему:
- защита от случайных/аномальных правок, при этом память всё ещё обучается на стабильных корректировках.

### 7) Миграционная стратегия
Решение:
- для ускоренной разработки выполнять destructive migration всего SwiftData store;
- после reset память инициализируется пустой, без seed/импорта.

Почему:
- снижает стоимость итерации на ранней стадии и убирает риски частичной несовместимости схем.

Альтернативы:
- мягкая миграция с переносом старых данных: безопаснее для прода, но сложнее в поддержке в текущей фазе.

## Risks / Trade-offs

- [Шумные подсказки при большом числе похожих логов] → Ограничение окна до 50, максимум 5 подсказок, ранжирование частота+свежесть.
- [Ложные гипотезы предпочтений] → Хранение как hypothesis + подтверждение после 2 повторов.
- [Расхождение текста и структуры] → Явный флаг `out_of_sync` + default-ветка «Переоценить по тексту» + сброс структуры при очистке текста.
- [Персонализация на неполных данных] → `low confidence` + `data-gap` + выборочные уточнения.
- [Рост объёма памяти] → Retention и очистка устаревших гипотез/шаблонов.
- [Потеря данных из-за destructive migration] → Приемлемо на текущей стадии разработки без продовых пользователей.

## Migration Plan

1. Добавить новые сущности/поля SwiftData для памяти, источника структуры и состояния `out_of_sync`.
2. Выполнить destructive reset всего SwiftData store.
3. Запустить приложение с пустой памятью и новой схемой.
4. Включить новый UI-поток подсказок, mini-уточнений порции и inline-подтверждений гипотез.
5. Включить мягкое применение памяти в агентском контуре (без блокирующих переходов).
6. Добавить контроль памяти (точечное удаление/сброс) и telemetry/debug поля для проверки корректности.
7. Верифицировать сценарии повторного наполнения памяти с нуля.

Rollback strategy:
- при критической проблеме выполнить повторный reset store и вернуться к предыдущей версии схемы/ветки.
