## ADDED Requirements

### Requirement: Учёт памяти в рекомендациях
Система SHALL учитывать мягкие предпочтения, подтверждённые гипотезы и открытые `data-gap` при формировании советов.

#### Scenario: Рекомендация с учётом предпочтения
- **WHEN** у пользователя подтверждено мягкое предпочтение
- **THEN** рекомендации приоритизируют варианты, не конфликтующие с этим предпочтением

### Requirement: Прозрачность влияющих факторов
Система SHALL показывать explainability рекомендаций в раскрываемом блоке «Почему этот совет», с top-2 факторов по умолчанию и возможностью показа полного списка.

#### Scenario: Объяснение рекомендаций
- **WHEN** пользователь получает совет
- **THEN** система показывает блок «Почему этот совет» с двумя ключевыми факторами и действием «Показать всё»

#### Scenario: Раскрытие полного списка факторов
- **WHEN** пользователь нажимает «Показать всё»
- **THEN** система отображает полный список сигналов памяти, учтённых в рекомендации

### Requirement: Учёт неполных данных
Система SHALL снижать уверенность рекомендаций при наличии открытых `data-gap`, предлагать точечные уточнения и объяснять, почему недостающие поля важны для точности.

#### Scenario: Рекомендация при открытых data-gap
- **WHEN** есть незакрытые `data-gap` по частым блюдам
- **THEN** рекомендация формируется с пометкой о неполноте данных, списком приоритетных уточнений и кратким пояснением влияния этих пробелов на точность

### Requirement: Конфликт подтверждённых предпочтений
Система SHALL при конфликте подтверждённого мягкого предпочтения со свежими логами снижать вес предпочтения и помечать его как `under_review` до переподтверждения.

#### Scenario: Конфликт предпочтения и новых данных
- **WHEN** подтверждённое предпочтение конфликтует с недавними повторяющимися логами пользователя
- **THEN** система помечает предпочтение как `under_review` и снижает его влияние в рекомендациях
