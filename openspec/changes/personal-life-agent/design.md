## Context

Приложение «персональный агент питания» — iOS-приложение на SwiftUI с локальным хранением данных и прямыми вызовами LLM API. Пользователь логирует приёмы пищи текстом, система через LLM структурирует описания в ингредиенты, считает калории и БЖУ через локальный справочник, даёт рекомендации и аналитику. Бэкенд отсутствует — все данные на устройстве, LLM-запросы идут напрямую от клиента с API-ключом пользователя.

## Goals / Non-Goals

**Goals:**
- Логирование питания с минимальным трением (текстовый ввод, один тап для сохранения)
- Автоматическая структуризация описаний через LLM и точный подсчёт БЖУ через локальный справочник
- Контекстные уведомления-напоминания по настраиваемым окнам питания
- Дневной LLM-анализ и on-demand консультации по питанию
- Аналитика питания в стиле Apple Health (день/неделя/месяц)
- Виджет для домашнего экрана с дневной сводкой

**Non-Goals:**
- Другие типы событий (тренировки, заметки, чекины) — отложены на будущее
- Голосовой ввод, фото, LLM Vision — отложены на будущее
- Бэкенд, облачная синхронизация, мультидевайс — не в MVP
- Социальные функции, шаринг, экспорт данных
- Push-уведомления через APNs (только локальные)

## Decisions

### 1. Архитектура данных: плоская модель SwiftData, две сущности

**Выбор**: две SwiftData-сущности со связью one-to-many:

- `MealEvent`: `windowLabel`, `timestamp`, `status` (empty / raw / structured), `rawText`, агрегированные `calories`, `proteins`, `fats`, `carbs`
- `Ingredient`: `name`, `amount`, `unit`, `calories`, `proteins`, `fats`, `carbs` — ссылается на родительский `MealEvent`

Три состояния `status`:
- `empty` — приём пищи не залогирован (серая карточка)
- `raw` — введён сырой текст, ждёт структуризации через LLM (жёлтая карточка)
- `structured` — ингредиенты извлечены, БЖУ посчитаны (зелёная карточка)

Осознанный пропуск приёма пищи («Не ел» в уведомлении) — отдельный `MealEvent` с `status = skipped`, нулевыми БЖУ. Отличается от пустого слота (нет данных) — важно для аналитики паттернов питания.

**Альтернатива (JSON-блоб)**: ингредиенты внутри JSON-поля. Отклонено — все поля понятные и конкретные, JSON не нужен.

**Альтернатива (строгие типы под каждый тип события)**: отклонено — MVP только питание.

### 2. Локальный справочник продуктов: SwiftData FoodItem, seed из JSON-бандла

**Выбор**: сущность `FoodItem` в SwiftData — `name`, `calories`, `proteins`, `fats`, `carbs` (на 100г). При первом запуске `ModelContainer` засевается из JSON-файла в бандле приложения (500–1000 продуктов: крупы, мясо, овощи, молочка, фрукты, полуфабрикаты). Пользователь может добавлять свои продукты через интерфейс — они хранятся в той же таблице.

LLM структурирует текст → возвращает нормализованные названия ингредиентов → приложение ищет точное или fuzzy-совпадение в `FoodItem` → умножает БЖУ на указанное количество → заполняет `Ingredient`.

**Альтернатива (LLM сама считает калории)**: LLM проставляет БЖУ в ответе. Отклонено — LLM галлюцинирует с цифрами, справочник даёт точность.

**Альтернатива (статический plist)**: файл в ресурсах без SwiftData. Отклонено — нельзя расширять пользователем, нет query-возможностей.

### 3. LLM-интеграция: прямые запросы, OpenAI-совместимый API

**Выбор**: единый клиент для OpenAI Chat Completions API. Настройки как в Cursor: пользователь вводит endpoint, API-ключ, модель (текстовые поля). Два LLM-сценария:

1. **Структуризация** — текст пользователя → список ингредиентов `[{name, amount, unit}]`. Вызывается при логировании (жёлтая → зелёная) или батчем для накопившихся жёлтых событий.
2. **Совет дня** — все зелёные приёмы за день → сводка с ингредиентами и БЖУ → качественный анализ питания (баланс, клетчатка, сахар, фастфуд, рекомендации).

Рекомендация на следующий приём — вкладка на карточке слота. Не полноценный LLM-вызов, а часть совета дня: система показывает ранее полученные рекомендации.

**Альтернатива (3 отдельных сценария)**: структуризация, совет дня, on-demand консультация. Отклонено для MVP — консультация сводится к рекомендации, которая уже часть совета дня.

**Альтернатива (бэкенд-прокси)**: сервер-посредник. Отклонено — требование локальности данных и отсутствия бэкенда.

### 4. Пайплайн обработки: немедленная структуризация + fallback-батч

**Выбор**: при сохранении сырого текста (статус `raw`) запускается `Task` → LLM-структуризация → поиск ингредиентов в `FoodItem` → создание `Ingredient`-записей → агрегация БЖУ в `MealEvent` → статус `structured`. При отсутствии сети или ошибке событие остаётся `raw`. Кнопка «Обработать всё» на дашборде и per-card кнопка для повторных попыток.

**Две фазы дневного анализа**:
1. Все приёмы должны стать `structured` (зелёными) — через автоматическую или ручную структуризацию
2. Кнопка «Совет дня» внизу дашборда, активна когда все слоты зелёные → отправка полной структуры за день → качественный анализ

**Фоновый запрос**: `UIApplication.beginBackgroundTask` для завершения активного LLM-запроса при сворачивании приложения (~30 секунд).

**Альтернатива (BGTaskScheduler)**: фоновая обработка. Отклонено — ненадёжно в iOS, запуск не гарантирован.

**Альтернатива (только батч)**: обработка только вечером. Отклонено — пользователь не видит результата сразу.

### 5. Уведомления: локальные пуши, 3 действия, фиксированные окна

**Выбор**: `UNUserNotificationCenter` + `UNNotificationAction` (3 кнопки: «Записать» — открывает лог, «Не ел» — фиксирует skipped, «Напомнить позже» — повтор через 30 минут). Окна питания настраиваются пользователем (до 6, одинаковые на все дни недели). При закрытии окна без лога — пуш.

**Анализ дня**: запускается по заполнению всех окон, не по фиксированному времени. Если последнее окно закрыто и все зелёные — кнопка «Совет дня» активна. Если окно закрыто, но не все слоты заполнены — локальный пуш с напоминанием заполнить. Если совет дня пропущен (не вызван) — напоминание позже.

**Skipped vs пустой слот**: «Не ел» в уведомлении создаёт `MealEvent` со статусом `skipped` — осознанный пропуск. Пустой слот (игнорирование уведомления) — отсутствие данных. Для аналитики это разные паттерны.

**Альтернатива (адаптивные окна)**: система учит паттерны и подстраивает время. Отклонено для MVP — требует накопленной истории.

### 6. Дашборд: DisclosureGroup со слотами и трёхцветная система

**Выбор**: главный экран — `DisclosureGroup` «Питание» с карточками-слотами (Завтрак, Обед, Ужин, Перекусы — до 6, настраиваются). Три цвета:
- Серая — `empty`, приём не залогирован
- Жёлтая — `raw`, введён текст, ждёт LLM-структуризации (на карточке отображается `rawText`)
- Зелёная — `structured`, ингредиенты извлечены, БЖУ посчитаны (карточка показывает сводку: блюда, калории, БЖУ)

Skipped-карточка — серая с иконкой перечёркивания.

**Рекомендация — вкладка на карточке слота**: каждый незаполненный слот имеет вкладку «Рекомендация», где LLM показывает ориентир (калории, распределение) на оставшиеся приёмы, без конкретного меню. Живёт отдельно от лога.

**Кнопка «Совет дня»** внизу дашборда, активна когда все слоты зелёные. Отправляет полную структуру за день в LLM → качественный анализ (баланс, сахар, фастфуд, рекомендации).

**Структурированный ввод** без LLM: на карточке логирования над полем ввода три быстрых действия — «Из истории» (последние 3–5 приёмов), «Из справочника» (поиск по FoodItem), «Шаблоны» (сохранённые пользователем). Выбор из любого источника → сразу статус `structured`.

**Альтернатива (чат-интерфейс)**: лента сообщений. Отклонено пользователем — неудобно для ежедневного трекинга питания.

### 7. Вкладка аналитики: Apple Health стиль

**Выбор**: вкладка «Аналитика» с переключателем день/неделя/месяц. Под ним карточки метрик: калории, белки, жиры, углеводы, дефицит. Каждая метрика кликабельна — открывает детальный экран с графиком (Swift Charts) за выбранный период.

**Альтернатива (вся аналитика на дашборде)**: скролл вниз на главном экране. Отклонено — дашборд перегружается, быстрый доступ теряется.

### 8. Onboarding: минимальный, 2 минуты до первого лога

**Выбор**: 3 шага:
1. API-ключ, endpoint, модель (настройки LLM как в Cursor)
2. Цель — переключатель: простой ввод калорий ИЛИ калькулятор (пол, возраст, рост, вес, уровень активности). Формула Mifflin-St Jeor: BMR × коэффициент активности = TDEE, от TDEE — здоровый дефицит 10–15%.
3. Окна питания — 3 стандартных (завтрак 7:00–10:00, обед 12:00–15:00, ужин 18:00–21:00), можно редактировать.

Всё остальное — в настройках.

**Альтернатива (wizard с 5+ шагами)**: пошаговый мастер со всеми настройками. Отклонено — время до первого лога критично.

### 9. Виджет: WidgetKit, дневная сводка

**Выбор**: medium-виджет с прогресс-барами по калориям и БЖУ за сегодня, индикатор заполненных окон питания. Обновление через `TimelineProvider` раз в 15 минут или по изменению данных.

### 10. Настройки: три раздела

**Выбор**: вкладка «Настройки» с тремя секциями:
- **LLM**: endpoint, API-ключ, модель (текстовые поля, как в Cursor)
- **Цель**: переключатель «Калории» (простое число) / «Калькулятор» (параметры: пол, возраст, рост, вес, активность, формула Mifflin-St Jeor)
- **Окна питания**: список до 6 записей, каждая — название + время начала/конца. Окна одинаковые на все дни недели. Добавление через кнопку «+»

## Risks / Trade-offs

- **[Точность LLM-структуризации]** — LLM может неверно извлечь ингредиенты или оценить порцию → Справочник FoodItem даёт точные цифры на 100г, LLM отвечает только за нормализацию текста. Карточка показывает извлечённые ингредиенты — пользователь видит и может поправить.
- **[Стоимость API]** — каждый вызов LLM платный для пользователя → батч-обработка нескольких событий за один запрос. Пользователь видит прогресс-бар и данные перед отправкой. Выбор дешёвых моделей (DeepSeek, GPT-4o-mini).
- **[Офлайн-режим]** — без интернета LLM недоступна → события сохраняются жёлтыми, кнопка «Обработать» для повторной попытки при появлении сети. Функции аналитики (подсчёт калорий из справочника) работают офлайн после структуризации.
- **[Потеря данных]** — все данные только на устройстве → в будущем добавить экспорт/импорт (JSON). Для MVP — информирование пользователя об отсутствии облачного бэкапа.
- **[Конфиденциальность]** — API-ключ и данные о питании хранятся локально → ключ в Keychain, данные только в SwiftData, запросы уходят напрямую выбранному провайдеру.
