## Context

Приложение «персональный агент питания» — iOS-приложение на SwiftUI с локальным хранением данных и прямыми вызовами LLM API. Пользователь логирует приёмы пищи текстом, система через LLM структурирует описания в ингредиенты, считает калории и БЖУ через локальный справочник, даёт рекомендации и аналитику. Бэкенд отсутствует — все данные на устройстве, LLM-запросы идут напрямую от клиента с API-ключом пользователя.

## Goals / Non-Goals

**Goals:**
- Логирование питания с минимальным трением (текстовый ввод, один тап для сохранения)
- Автоматическая структуризация описаний через LLM и точный подсчёт БЖУ через локальный справочник
- Контекстные уведомления-напоминания по настраиваемым окнам питания
- Дневной LLM-анализ и on-demand консультации по питанию
- Аналитика питания в стиле Apple Health (день/неделя/месяц)
- Виджет для домашнего экрана с дневной сводкой

**Non-Goals:**
- Другие типы событий (тренировки, заметки, чекины) — отложены на будущее
- Голосовой ввод, фото, LLM Vision — отложены на будущее
- Бэкенд, облачная синхронизация, мультидевайс — не в MVP
- Социальные функции, шаринг, экспорт данных
- Push-уведомления через APNs (только локальные)

## Decisions

### 1. Архитектура данных: гибридная модель SwiftData

**Выбор**: единая сущность `MealEvent` с индексируемыми полями (`calories`, `proteins`, `fats`, `carbs`, `windowLabel`, `timestamp`, `processed`) и JSON-полем `payload` для гибких данных (ингредиенты, сырой текст).

**Альтернатива (строгие типы)**: отдельные SwiftData-модели под каждый тип события. Отклонено — избыточно для MVP, где пока только питание.

**Альтернатива (чистый JSON-блоб)**: всё в одном `Data`-поле. Отклонено — невозможно индексировать и query по калориям и датам без парсинга.

### 2. Локальный справочник продуктов: SwiftData FoodItem с seed-данными

**Выбор**: сущность `FoodItem` в SwiftData, предзаполненная seed-данными при первом запуске через `ModelContainer` с кастомной конфигурацией. Пользователь может расширять справочник своими продуктами.

**Альтернатива (JSON/plist в бандле)**: статический файл в ресурсах. Отклонено — нельзя расширять, нет query-возможностей SwiftData.

### 3. LLM-интеграция: прямые запросы, OpenAI-совместимый API

**Выбор**: единый клиент для OpenAI Chat Completions API. Пользователь вводит endpoint, API-ключ и модель. Приложение формирует промпты для трёх сценариев: структуризация описания, дневной анализ, on-demand консультация.

**Альтернатива (отдельные клиенты под каждого провайдера)**: по одному клиенту на OpenAI, DeepSeek, Groq. Отклонено — все они совместимы с OpenAI API, дублирование кода неоправданно.

**Альтернатива (бэкенд-прокси)**: сервер-посредник. Отклонено — требование локальности данных и отсутствия бэкенда.

### 4. Пайплайн обработки: немедленная структуризация + fallback-батч

**Выбор**: при сохранении текста запускается `Task` → LLM-запрос на структуризацию → поиск ингредиентов в `FoodItem` → подсчёт БЖУ → обновление `MealEvent` (жёлтая → зелёная). При отсутствии сети или ошибке событие остаётся жёлтым. Кнопка «Обработать всё» или per-card кнопка для повторных попыток.

**Альтернатива (только батч)**: обработка только вечером. Отклонено — пользователь не видит результата сразу, UX страдает.

**Альтернатива (BGTaskScheduler)**: фоновая обработка. Отклонено — ненадёжно в iOS, запуск не гарантирован.

**Защита фонового запроса**: `UIApplication.beginBackgroundTask` для завершения активного LLM-запроса при сворачивании приложения (~30 секунд).

### 5. Уведомления: локальные пуши, 3 действия, фиксированные окна

**Выбор**: `UNUserNotificationCenter` + `UNNotificationAction` (3 кнопки: «Записать», «Не ел», «Напомнить позже»). Окна питания настраиваются пользователем (до 6). При закрытии окна без лога — пуш. При нажатии «Не ел» — фиксируется пропуск. «Напомнить позже» — повтор через 30 минут. Локальные пуши также используются для приглашения на дневной анализ.

**Альтернатива (адаптивные окна)**: система учит паттерны и подстраивает время. Отклонено для MVP — требует накопленной истории.

### 6. Дашборд: DisclosureGroup со слотами и трёхцветная система

**Выбор**: главный экран — `DisclosureGroup` «Питание» с карточками-слотами (Завтрак, Обед, Ужин, Перекусы). Три цвета: серая (пусто), жёлтая (сырые данные, ждёт LLM), зелёная (структурировано с БЖУ). Под заполненным слотом — кнопка «Получить рекомендацию» для следующего приёма. Кнопка «Анализ дня» внизу дашборда, активна при заполнении всех окон.

**Альтернатива (чат-интерфейс)**: лента сообщений. Отклонено пользователем — неудобно для ежедневного трекинга питания.

### 7. Вкладка аналитики: Apple Health стиль

**Выбор**: вкладка «Аналитика» с переключателем день/неделя/месяц. Под ним карточки метрик: калории, белки, жиры, углеводы, дефицит. Каждая метрика кликабельна — открывает детальный экран с графиком (Swift Charts) за выбранный период.

**Альтернатива (вся аналитика на дашборде)**: скролл вниз на главном экране. Отклонено — дашборд перегружается, быстрый доступ теряется.

### 8. Onboarding: минимальный, 2 минуты до первого лога

**Выбор**: 3 шага — API-ключ, цель (калории: простой ввод или калькулятор с параметрами), 3 стандартных окна питания (завтрак/обед/ужин с возможностью редактирования). Всё остальное — в настройках.

**Альтернатива (wizard с 5+ шагами)**: пошаговый мастер со всеми настройками. Отклонено — время до первого лога критично.

### 9. Виджет: WidgetKit, дневная сводка

**Выбор**: medium-виджет с прогресс-барами по калориям и БЖУ за сегодня, индикатор заполненных окон питания. Обновление через `TimelineProvider` раз в 15 минут или по изменению данных.

## Risks / Trade-offs

- **[Точность LLM-структуризации]** — LLM может неверно извлечь ингредиенты или оценить порцию → Справочник FoodItem даёт точные цифры на 100г, LLM отвечает только за нормализацию текста. Карточка показывает извлечённые ингредиенты — пользователь видит и может поправить.
- **[Стоимость API]** — каждый вызов LLM платный для пользователя → батч-обработка нескольких событий за один запрос. Пользователь видит прогресс-бар и данные перед отправкой. Выбор дешёвых моделей (DeepSeek, GPT-4o-mini).
- **[Офлайн-режим]** — без интернета LLM недоступна → события сохраняются жёлтыми, кнопка «Обработать» для повторной попытки при появлении сети. Функции аналитики (подсчёт калорий из справочника) работают офлайн после структуризации.
- **[Потеря данных]** — все данные только на устройстве → в будущем добавить экспорт/импорт (JSON). Для MVP — информирование пользователя об отсутствии облачного бэкапа.
- **[Конфиденциальность]** — API-ключ и данные о питании хранятся локально → ключ в Keychain, данные только в SwiftData, запросы уходят напрямую выбранному провайдеру.
