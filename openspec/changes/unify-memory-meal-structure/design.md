## Context

Текущий поток оценки питания использует два источника данных (LLM и память), но результат из memory-path может приходить в менее структурированном виде, чем результат LLM-path. Из-за этого `MealEvent.items` заполняется неравномерно, а UI-функции, завязанные на позиционную структуру (подсветка калорийных позиций, компонентный анализ), работают нестабильно.

Ограничения и предпосылки:
- Приложение на активной pre-beta стадии, допускается destructive reset данных.
- Пользовательский ввод остаётся свободным текстом.
- Каноническая структура должна быть одинаковой независимо от источника.
- В рамках change не перерабатывается система советов (только данные и подсветка).

Стейкхолдеры:
- Пользователь (получает предсказуемый состав блюда и корректную подсветку).
- Разработка (получает единый pipeline без ветвлений и дублирующей логики).

## Goals / Non-Goals

**Goals:**
- Зафиксировать `MealEvent.items` как единый канон данных для состава блюда.
- Выровнять memory-path и llm-path в единый pipeline сохранения события.
- Ввести обязательный fallback в LLM при неполных/битых memory-items.
- Сделать подсветку калорийных позиций детерминированной по правилу:
  - `totalKcal >= 120`
  - `kcalPer100g >= 250`
  - `grams >= 10`
- Упростить поддержку через destructive reset без миграций совместимости.

**Non-Goals:**
- Переработка текста советов и стратегии рекомендаций.
- Изменение UX свободного текстового ввода на принудительный ингредиентный ввод.
- Backfill/миграция старых записей в новый формат.

## Decisions

### 1) Единый канон: `MealEvent.items`
- Решение: хранить только канонический itemized-состав в `MealEvent.items`; UI «Состав блюда» читает только его.
- Почему: исключает расхождения между слоями и упрощает аналитику.
- Альтернативы:
  - Отдельная каноническая структура в памяти с проекцией в UI — отклонено из-за лишней сложности и риска рассинхрона.

### 2) Один pipeline без развилок
- Решение: `text -> (memory items валидны ? use : LLM) -> canonical items -> save`.
- Почему: одинаковый выходной формат и предсказуемое поведение для всех downstream-функций.
- Альтернативы:
  - Разные pipeline для memory/LLM с последующей склейкой — отклонено из-за роста ветвлений и edge-cases.

### 3) Политика памяти: хранить и фразу, и нормализованные items
- Решение: память содержит исходную фразу (для UX/истории) и нормализованные items (для повторного использования), но источник истины всегда `items`.
- Почему: сохраняет удобство подсказок без потери структурного канона.
- Альтернативы:
  - Хранить только текст — отклонено (слабая повторная применимость).
  - Хранить только items — отклонено (хуже UX для подсказок в поле ввода).

### 4) Fallback при неполных memory-items
- Решение: если memory-items неполные/битые, отправлять весь приём пищи в LLM для полной реконструкции, частично битые items не сохранять.
- Почему: избегаем накопления неканоничных событий и «тихих» искажений аналитики.
- Альтернативы:
  - Частично принимать валидные items — отклонено (риск искажения структуры).
  - Блокировать сохранение — отклонено (хуже UX).

### 5) Метаданные источника
- Решение: хранить источник (`memory`/`llm`) в metadata события, но не показывать на уровне item в основном UI.
- Почему: сохраняем отладочную прозрачность без UI-шума.

### 6) Подсветка калорийных позиций по тройному правилу
- Решение: подсвечивать позицию только при выполнении всех трёх условий: `totalKcal >= 120`, `kcalPer100g >= 250`, `grams >= 10`.
- Почему: учитывается и калорийная плотность, и фактический вклад порции; уменьшается шум на малых добавках.
- Альтернативы:
  - Только абсолютный порог ккал — отклонено (теряется плотность).
  - Top-N по блюду — отклонено (пользователь просит подсвечивать все реально калорийные позиции).

### 7) BREAKING стратегия данных
- Решение: destructive reset хранилища без data migration.
- Почему: проект ранний, это сокращает сложность и ускоряет стабилизацию нового контракта.
- Альтернативы:
  - Миграция совместимости — отклонено как избыточная для текущей стадии.

### 8) Триггер и хранение состояния reset
- Решение: one-shot auto-reset по версии схемы на старте приложения; флаг применённой версии хранится в `UserDefaults`.
- Почему: исключается ручной шаг, снижается риск запуска на полу-совместимых данных.
- Альтернативы:
  - Ручной reset через dev-кнопку/флаг — отклонено как менее надёжный.
  - Персистить флаг в доменной модели — отклонено как лишнее усложнение.

### 9) Ошибка fallback LLM
- Решение: если fallback в LLM завершился ошибкой, событие не сохраняется (жёсткий fail).
- Почему: сохраняется требование единого канона без частичных/невалидных сохранений.
- Альтернативы:
  - Сохранять только текст/черновик — отклонено, чтобы не плодить неконсистентные записи.
  - Сохранять частично валидные memory items — отклонено из-за риска искажений.

### 10) Явный пересчёт только по кнопке
- Решение: редактирование текста само по себе не запускает LLM; пересчёт выполняется только по явному действию пользователя.
- Почему: соответствует выбранному UX-поведению и исключает неожиданные фоновые переоценки.
- Альтернативы:
  - Автопересчёт при каждом изменении текста — отклонено.

## Risks / Trade-offs

- [Риск] Временная потеря существующих данных из-за destructive reset.  
  → Mitigation: явно зафиксировать BREAKING в артефактах и выполнить reset единоразово в релевантной ветке.

- [Риск] Дополнительные LLM-вызовы при низком качестве memory-items.  
  → Mitigation: валидировать полноту memory-items до вызова LLM и переиспользовать валидные канонические items.

- [Риск] Порог подсветки может потребовать тюнинга на реальных сценариях.  
  → Mitigation: держать правило централизованным и легко параметризуемым для следующего change.

- [Риск] Мягкая локальная чистка названий может скрывать ошибки контракта LLM.  
  → Mitigation: логировать случаи чистки для последующей корректировки prompt/контракта.

- [Риск] Жёсткий fail при ошибке fallback может приводить к потере ввода в текущей сессии.  
  → Mitigation: показывать пользователю ошибку пересчёта и оставлять текст в редакторе до явного повтора/отмены.

## Migration Plan

1. Обновить модели/контракты pipeline под единый канонический itemized-выход.
2. Реализовать в runtime валидацию memory-items и fallback в LLM при неполноте.
3. Применить правило подсветки калорийных позиций в едином месте расчёта UI-статуса.
4. Выполнить destructive reset локального хранилища (SwiftData/SQLite) без data migration.
5. Реализовать one-shot механизм reset по версии схемы с флагом в `UserDefaults` (без пользовательского уведомления).
6. Провести smoke-проверку сценариев:
   - memory-path с валидными items,
   - memory-path с неполными items (fallback в LLM),
   - корректная подсветка по тройному правилу,
   - изменение текста без явного пересчёта,
   - ошибку fallback LLM (событие не сохраняется).

Rollback:
- Технически возможен только откатом кода до change и повторным reset, так как обратной миграции данных нет.

## Text–Structure Sync State Machine

### Состояния
- `Synced`: текст и `MealEvent.items` согласованы относительно baseline.
- `Outdated`: текст семантически изменён относительно baseline, структура потенциально неактуальна.
- `PendingRecalc`: пользователь инициировал явный пересчёт (LLM) после состояния `Outdated`.
- `RecalcFailed`: попытка пересчёта завершилась ошибкой; сохраняется прежний канон и `Outdated=true`.

### Данные состояния
- `baselineTextNormalized`: normalized-текст, по которому последний раз успешно подтверждена структура.
- `structureOutdated: Bool`: флаг рассинхрона на уровне `MealEvent`.
- `structureLastSyncedAt: Date?`: timestamp последнего успешного подтверждения синхронизации (служебное metadata-поле, не отображается в UI).

### Нормализация текста
Перед сравнением текста применяется простая нормализация:
- `trim` по краям;
- схлопывание повторных пробелов до одного.

Изменение только пробелов, не меняющее normalized-строку, не переводит событие в `Outdated`.

### Переходы
1. Инициализация нового события:
- До появления первой валидной структуры состояние синхронизации не применяется.
- После первого успешного построения `items` (из memory или LLM):
  - `baselineTextNormalized = normalize(currentText)`
  - `structureOutdated = false`
  - `state = Synced`

2. Редактирование текста:
- Если `normalize(currentText) != baselineTextNormalized` → `structureOutdated = true`, `state = Outdated`.
- Если строки равны → состояние не меняется.

3. Явный пересчёт по кнопке:
- Если `state = Outdated` и `normalize(currentText) == baselineTextNormalized`, LLM-вызов пропускается (no-op).
- Если `state = Outdated` и текст действительно изменился, выполняется LLM пересчёт:
  - Успех: обновить `MealEvent.items`, `baselineTextNormalized`, `structureOutdated=false`, `structureLastSyncedAt=now`, `state = Synced`.
  - Ошибка: оставить текущие `items` без изменений, `structureOutdated=true`, `state = RecalcFailed`; если это fallback перед сохранением нового события — сохранение не выполнять.

4. Ручная правка `items`:
- Не сбрасывает `structureOutdated`.
- Не обновляет `baselineTextNormalized`.
- Только успешный LLM-пересчёт может подтвердить повторную синхронизацию.

### Правила сохранения
- Событие можно сохранять и в `Outdated`, сохраняя текущие `items` как канон и текущий текст как пользовательскую запись.
- `structureOutdated` и `structureLastSyncedAt` персистятся на уровне `MealEvent` для последующей диагностики и поведения UI.
