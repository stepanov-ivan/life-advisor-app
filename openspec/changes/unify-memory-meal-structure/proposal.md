## Why

Сейчас события, собранные из памяти, могут сохраняться в форме, которая расходится с каноническим `MealEvent.items`, из-за чего ухудшается прозрачность анализа и некорректно работает подсветка калорийных позиций. Нужно привести все источники к единому каноническому формату состава блюда, чтобы аналитика и UI работали предсказуемо.

## What Changes

- Зафиксировать единый канон: `MealEvent.items` — единственный источник истины для состава блюда и отображения секции «Состав блюда» в UI.
- Ввести один обязательный pipeline без развилок: `text -> (валидные memory items ? use : LLM) -> canonical items -> save`.
- Изменить memory-path: память хранит и исходную фразу, и нормализованные позиции, но при расчётах/анализе истина всегда за `items`.
- Добавить fallback-правило: при неполных/битых memory-items выполнять полную реконструкцию структуры через LLM перед сохранением.
- Зафиксировать поведение синхронизации текста и структуры: изменение нормализованного текста помечает структуру как неактуальную (`structureOutdated=true`), а сброс этого статуса возможен только после успешного явного пересчёта через LLM.
- Зафиксировать правило подсветки калорийных позиций: подсвечивать позицию, если одновременно `totalKcal >= 120`, `kcalPer100g >= 250` и `grams >= 10`.
- Выполнить **BREAKING** destructive reset хранилища/памяти под новый формат без миграции старых записей.
- Ограничить scope change: только канон данных и корректная подсветка; переработка советов выносится в отдельный change.

## Capabilities

### New Capabilities
- `meal-structure-normalization`: Унификация состава блюда в канонический itemized-формат независимо от источника (memory/LLM) с едиными правилами сохранения.

### Modified Capabilities
- `nutrition-memory`: Память должна хранить нормализованные позиции для прямого использования в каноническом pipeline; при неполных данных обязателен fallback в LLM.
- `event-logging`: Логирование приёма пищи должно сохранять событие только в едином структурированном виде через один pipeline и персистить metadata синхронизации (`structureOutdated`, `structureLastSyncedAt`).
- `llm-agent`: Агентный путь должен гарантировать канонический itemized-результат перед сохранением события и поддерживать fallback-реконструкцию.

## Impact

- Затронутые системы: pipeline оценки и сохранения приёма пищи, слой памяти питания, подсветка калорийных позиций в UI.
- Затронутый код: runtime обработки memory/LLM результата, нормализация `MealEvent.items`, логика подсветки калорийных позиций.
- Затронутый код: runtime обработки memory/LLM результата, нормализация `MealEvent.items`, metadata синхронизации текста/структуры, логика подсветки калорийных позиций.
- Данные/хранилище: destructive reset без data migration.
- UX: пользовательский ввод остаётся свободным текстом; каноническая структура формируется/поддерживается системой и используется как база анализа.
