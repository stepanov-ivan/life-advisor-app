## ADDED Requirements

### Requirement: Канонический состав блюда
Система SHALL сохранять состав блюда в едином каноническом формате `MealEvent.items` независимо от источника данных (LLM или память).

#### Scenario: Канон после оценки из LLM
- **WHEN** пользователь сохраняет приём пищи после LLM-оценки
- **THEN** система сохраняет `MealEvent.items` как канонический список позиций для UI и аналитики

#### Scenario: Канон после использования памяти
- **WHEN** пользователь сохраняет приём пищи по memory-path
- **THEN** система сохраняет `MealEvent.items` в том же формате, что и для LLM-path, без отдельной ветки хранения

### Requirement: Единый pipeline сохранения
Система SHALL использовать один pipeline сохранения события: `text -> (валидные memory items ? use : LLM) -> canonical items -> save`.

#### Scenario: Валидные memory items
- **WHEN** memory items валидны и соответствуют каноническому контракту
- **THEN** система использует их напрямую для `MealEvent.items` и сохраняет событие без LLM-вызова

#### Scenario: Невалидные memory items
- **WHEN** memory items неполные или битые
- **THEN** система выполняет полную реконструкцию структуры через LLM до сохранения события

### Requirement: Правило подсветки калорийных позиций
Система SHALL подсвечивать калорийные позиции по единому правилу для текущего события.

#### Scenario: Позиция считается калорийной
- **WHEN** у позиции одновременно выполняются условия `totalKcal >= 120`, `kcalPer100g >= 250` и `grams >= 10`
- **THEN** система помечает позицию как калорийную в UI

#### Scenario: Позиция не проходит тройной порог
- **WHEN** хотя бы одно из условий тройного порога не выполняется
- **THEN** система не подсвечивает позицию как калорийную
