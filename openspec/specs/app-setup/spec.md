## Requirements
### Requirement: Onboarding — настройка LLM
При первом запуске система SHALL провести пользователя через onboarding из трёх шагов. Первый шаг — ввод endpoint, API-ключа и модели LLM (текстовые поля, как в Cursor).

#### Scenario: Успешный ввод LLM-конфигурации
- **WHEN** пользователь на первом шаге вводит endpoint `https://api.openai.com/v1`, ключ `sk-...` и модель `gpt-4o-mini`, нажимает «Далее»
- **THEN** ключ сохраняется в Keychain, endpoint и модель — в UserDefaults, переход ко второму шагу

#### Scenario: Пропуск LLM-конфигурации
- **WHEN** пользователь нажимает «Настрою позже» на шаге LLM
- **THEN** onboarding продолжается без сохранения ключа, при попытке LLM-запроса в будущем система предложит настроить провайдера

### Requirement: Onboarding — цель по калориям
Второй шаг onboarding SHALL предложить переключатель: простой ввод целевых калорий ИЛИ калькулятор на основе параметров тела.

#### Scenario: Простой ввод калорий
- **WHEN** пользователь выбирает «Калории», вводит 2000, нажимает «Далее»
- **THEN** цель 2000 ккал сохраняется, переход к третьему шагу

#### Scenario: Калькулятор калорий
- **WHEN** пользователь выбирает «Калькулятор», заполняет: пол (мужской), возраст (30), рост (180 см), вес (80 кг), активность (средняя)
- **THEN** по формуле Mifflin-St Jeor вычисляется TDEE (~2800 ккал), от него здоровый дефицит 10–15% (~2400 ккал), цель сохраняется

### Requirement: Onboarding — окна питания
Третий шаг onboarding SHALL предлагать 3 стандартных окна питания с фиксированными stable ids и локализованными display names. Система SHALL сохранять логику окон независимо от локализованного текста и создавать новые системные окна на текущем effective language.

#### Scenario: Стандартные окна приняты на английском языке
- **WHEN** effective language приложения английский и пользователь на onboarding принимает стандартные окна питания
- **THEN** создаются 3 `MealWindow`-записи с фиксированными идентификаторами `breakfast`, `lunch`, `dinner`, а в UI отображаются локализованные названия `Breakfast`, `Lunch`, `Dinner`

#### Scenario: Стандартные окна приняты
- **WHEN** пользователь видит окна: Завтрак (7:00–10:00), Обед (12:00–15:00), Ужин (18:00–21:00) и нажимает «Начать»
- **THEN** создаются 3 `MealWindow`-записи, onboarding завершается, открывается дашборд

#### Scenario: Редактирование времени стандартных окон
- **WHEN** пользователь меняет время окна `breakfast` на 8:00–11:00 и окна `dinner` на 19:00–22:00
- **THEN** сохраняются изменённые интервалы окон без зависимости от локализованного display name

### Requirement: Настройки — раздел LLM
Вкладка «Настройки» SHALL содержать раздел LLM с полями: endpoint, API-ключ, модель.

#### Scenario: Изменение LLM-конфигурации
- **WHEN** пользователь меняет модель с `gpt-4o-mini` на `deepseek-chat`
- **THEN** настройка сохраняется, последующие запросы используют новую модель

### Requirement: Настройки — раздел Цель
Вкладка «Настройки» SHALL содержать раздел «Цель» с переключателем между простым вводом калорий и калькулятором.

#### Scenario: Переход с простого ввода на калькулятор
- **WHEN** пользователь переключает «Калории» → «Калькулятор» и заполняет параметры
- **THEN** цель пересчитывается по формуле Mifflin-St Jeor, отображается новое значение

#### Scenario: Ручная корректировка после калькулятора
- **WHEN** после расчёта калькулятора (2400 ккал) пользователь вручную меняет на 2200
- **THEN** цель = 2200 ккал, режим остаётся «Калории»

### Requirement: Настройки — раздел Окна питания
Вкладка «Настройки» SHALL содержать раздел «Окна питания» со списком системных окон питания, основанных на stable ids, с возможностью редактирования временных интервалов без зависимости поведения от локализованного текста окна.

#### Scenario: Отображение окон на русском языке
- **WHEN** effective language приложения русский и пользователь открывает раздел «Окна питания»
- **THEN** окна отображаются как локализованные русские названия, соответствующие их stable ids

#### Scenario: Отображение окон на английском языке
- **WHEN** effective language приложения английский и пользователь открывает раздел «Окна питания»
- **THEN** те же окна отображаются как английские локализованные названия при сохранении тех же идентификаторов и интервалов

#### Scenario: Добавление четвёртого окна
- **WHEN** пользователь нажимает «+», задаёт «Перекус» 16:00–17:00
- **THEN** новое окно добавляется в список, на дашборде появляется новый слот

#### Scenario: Удаление окна
- **WHEN** пользователь удаляет окно «Перекус»
- **THEN** окно удаляется из списка и с дашборда

#### Scenario: Превышение лимита окон
- **WHEN** у пользователя уже 6 окон, и он нажимает «+»
- **THEN** кнопка неактивна или показывает сообщение «Максимум 6 окон питания»

### Requirement: Настройки — выбор языка приложения
Вкладка «Настройки» SHALL содержать пользовательский выбор языка приложения в формате `System`, `English`, `Русский`.

#### Scenario: Пользователь выбирает системный язык
- **WHEN** пользователь выбирает `System` в настройках языка
- **THEN** приложение использует effective language, определённый по текущей системной локали устройства

#### Scenario: Пользователь выбирает русский язык вручную
- **WHEN** пользователь выбирает `Русский` в настройках языка
- **THEN** приложение сохраняет override языка и использует русский язык независимо от системной локали до следующего изменения настройки

### Requirement: Запуск приложения учитывает сохранённый язык
При запуске приложения система SHALL восстанавливать сохранённый язык приложения до отображения основных пользовательских экранов.

#### Scenario: Повторный запуск после выбора английского языка
- **WHEN** пользователь ранее выбрал `English`, завершил приложение и запускает его снова
- **THEN** основные экраны приложения открываются сразу на английском языке без необходимости повторного выбора
