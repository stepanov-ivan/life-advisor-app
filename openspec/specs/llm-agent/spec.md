## Purpose

Интеграция с LLM для структуризации текста питания в JSON-контракт с расширенными полями нутриентов.

## Requirements

### Requirement: Структуризация текста через LLM
Система SHALL отправлять текст логирования в LLM и получать строгий JSON-контракт оценки питания, где каждая позиция содержит чистое имя и порцию в граммах. Контракт расширен опциональными полями `saturated_fats`, `sugar`, `fiber`, `sodium`, `food_category`.

#### Scenario: Строгий контракт позиции
- **WHEN** LLM возвращает структурированный ответ
- **THEN** каждый item содержит обязательные поля `name` и `grams > 0`, без `unit`

#### Scenario: Чистое имя позиции
- **WHEN** LLM формирует item
- **THEN** поле `name` не содержит порционных маркеров, скобок и служебных обозначений (например, `~порция`, `3 шт`, `150г`)

#### Scenario: Невалидный контракт
- **WHEN** хотя бы один item нарушает контракт (`grams` отсутствует/невалиден или `name` грязный)
- **THEN** система помечает оценку как `parseFailed` целиком и не применяет частичный результат

### Requirement: Fallback реконструкции структуры
Система SHALL выполнять LLM-реконструкцию структуры как обязательный fallback при неполных memory-items и не сохранять событие при ошибке fallback.

#### Scenario: Fallback при неполных данных памяти
- **WHEN** memory-path не даёт валидный канонический набор позиций
- **THEN** система отправляет текст в LLM для полной реконструкции состава до сохранения события

#### Scenario: Ошибка fallback
- **WHEN** fallback-вызов LLM завершился ошибкой сети/парсинга
- **THEN** система не сохраняет событие и возвращает ошибку пользователю

### Requirement: Явный пересчёт по действию пользователя
Система SHALL пересчитывать структуру по отредактированному тексту только по явному действию пользователя.

#### Scenario: Редактирование текста без кнопки пересчёта
- **WHEN** пользователь изменил текст и не инициировал явный пересчёт
- **THEN** система не выполняет автоматический LLM-вызов и сохраняет текущую структуру

#### Scenario: Явный пересчёт без фактических изменений
- **WHEN** пользователь запрашивает пересчёт, но normalized-текст совпадает с baseline
- **THEN** система пропускает LLM-вызов как no-op и не обновляет `structureLastSyncedAt`

#### Scenario: Успешный явный пересчёт
- **WHEN** пользователь запрашивает пересчёт и LLM возвращает валидный результат
- **THEN** система обновляет канонический `MealEvent.items`, сбрасывает `structureOutdated` и обновляет `structureLastSyncedAt`

#### Scenario: Ошибка явного пересчёта
- **WHEN** пользователь запрашивает пересчёт и LLM возвращает ошибку
- **THEN** система сохраняет текущие `items` без изменений, оставляет `structureOutdated=true` и позволяет сохранить событие без пересчёта

### Requirement: Расширенный JSON-контракт LLM с полями нутриентов
Система SHALL расширить JSON-контракт LLM опциональными полями на уровне item и totals: `saturated_fats`, `sugar`, `fiber`, `sodium`, `food_category`.

#### Scenario: LLM возвращает все расширенные поля
- **WHEN** LLM успешно парсит приём и возвращает все новые поля
- **THEN** `EstimateItem` сохраняет все значения: saturated_fats, sugar, fiber, sodium, food_category

#### Scenario: LLM возвращает часть расширенных полей
- **WHEN** LLM возвращает только `sugar` и `fiber`, но не `saturated_fats`, `sodium`, `food_category`
- **THEN** отсутствующие поля сохраняются как nil/0, парсинг считается успешным

#### Scenario: Ни одно расширенное поле не возвращено
- **WHEN** LLM возвращает только базовые поля (calories, proteins, fats, carbs)
- **THEN** парсинг успешен, новые поля остаются пустыми, правила получают «нет данных»

### Requirement: Поле food_category с 8 значениями
Система SHALL принимать `food_category` как строку из фиксированного набора: `fruit`, `vegetable`, `whole_grain`, `legume`, `nut_seed`, `red_meat`, `processed_meat`, `other`.

#### Scenario: Валидная категория
- **WHEN** LLM возвращает `food_category: "whole_grain"`
- **THEN** значение сохраняется в `EstimateItem.foodCategory`

#### Scenario: Неизвестная категория
- **WHEN** LLM возвращает `food_category: "fish"` (не из списка)
- **THEN** значение заменяется на `"other"` и сохраняется

#### Scenario: Категория не указана
- **WHEN** LLM не возвращает `food_category`
- **THEN** `EstimateItem.foodCategory` = `nil`, presence-правила получают missing data

### Requirement: Единый промпт для всех полей
Система SHALL использовать один LLM-запрос для получения как базовых, так и расширенных полей без разделения на несколько вызовов.

#### Scenario: Успешный полный ответ
- **WHEN** LLM возвращает валидный JSON со всеми базовыми и расширенными полями
- **THEN** приём сохраняется с полными данными за один вызов

#### Scenario: Ошибка парсинга из-за базовых полей
- **WHEN** LLM возвращает невалидный ответ для базовых полей (calories, grams, name)
- **THEN** весь приём помечается `parseFailed`, расширенные поля не применяются
