## Requirements
### Requirement: Конфигурация LLM-провайдера
Пользователь SHALL иметь возможность настроить подключение к LLM-провайдеру, указав endpoint, API-ключ и модель. Настройки как в Cursor — три текстовых поля.

#### Scenario: Настройка провайдера в onboarding
- **WHEN** пользователь на первом шаге onboarding вводит endpoint `https://api.openai.com/v1`, API-ключ и модель `gpt-4o-mini`
- **THEN** настройки сохраняются в Keychain (ключ) и UserDefaults (endpoint, модель), приложение готово к LLM-запросам

#### Scenario: Смена провайдера в настройках
- **WHEN** пользователь меняет endpoint на `https://api.deepseek.com/v1` и модель на `deepseek-chat`
- **THEN** последующие LLM-запросы используют новый провайдер

### Requirement: Структуризация текста через LLM
Система SHALL отправлять сырой текст приёма пищи в LLM и получать нормализованный список ингредиентов.

#### Scenario: Успешная структуризация
- **WHEN** система отправляет текст «овсянка с ягодами и кофе с молоком» с промптом «извлеки ингредиенты в формате JSON: [{name, amount, unit}]»
- **THEN** LLM возвращает `[{name: "овсянка", amount: 100, unit: "г"}, {name: "ягоды", amount: 50, unit: "г"}, {name: "кофе", amount: 200, unit: "мл"}, {name: "молоко", amount: 50, unit: "мл"}]`

#### Scenario: Ошибка сети при структуризации
- **WHEN** LLM-запрос завершается ошибкой сети
- **THEN** `MealEvent` остаётся в статусе `raw`, пользователь видит кнопку «Обработать» на карточке

#### Scenario: Некорректный API-ключ
- **WHEN** LLM возвращает 401 Unauthorized
- **THEN** на карточке отображается красный индикатор «Ошибка авторизации», событие остаётся `raw`

#### Scenario: Таймаут запроса
- **WHEN** LLM-запрос превышает таймаут (30 секунд)
- **THEN** на карточке отображается ошибка «Превышено время ожидания», кнопка «Повторить»

### Requirement: Рекомендация на оставшиеся приёмы
Система SHALL предоставлять пользователю возможность запросить LLM-рекомендацию по распределению калорий и БЖУ на оставшиеся незаполненные окна питания.

#### Scenario: Запрос рекомендации
- **WHEN** пользователь залогировал завтрак (structured), открыл вкладку «Рекомендация» на слоте «Обед» и нажал «Получить рекомендацию»
- **THEN** система отправляет в LLM: съеденные приёмы с ингредиентами и БЖУ, цель, оставшиеся окна → LLM возвращает распределение калорий и БЖУ на оставшиеся приёмы с текстовыми рекомендациями

#### Scenario: Рекомендация без залогированных приёмов
- **WHEN** пользователь открывает вкладку «Рекомендация», но ни один приём ещё не залогирован
- **THEN** кнопка «Получить рекомендацию» неактивна с подсказкой «Заполните хотя бы один приём пищи»

#### Scenario: Кеширование рекомендации
- **WHEN** пользователь получает рекомендацию, а затем открывает вкладку «Рекомендация» на другом слоте
- **THEN** отображается последняя сохранённая рекомендация за сегодня, без повторного LLM-запроса. При новом логировании кеш сбрасывается

### Requirement: Совет дня через LLM
Система SHALL отправлять полную структуру приёмов пищи за день в LLM и получать качественный анализ питания. Все не-skipped окна должны быть в статусе `structured`. Skipped-слоты засчитываются и передаются в анализе как осознанно пропущенные.

#### Scenario: Успешный совет дня
- **WHEN** все не-skipped слоты зелёные, пользователь нажимает «Совет дня»
- **THEN** система отправляет JSON с ингредиентами и БЖУ по каждому приёму (включая skipped) + цель → LLM возвращает анализ: оценка баланса, содержание клетчатки, сахара, фастфуда, учёт пропущенных приёмов, рекомендации на завтра

#### Scenario: Совет дня с skipped-слотами
- **WHEN** пользователь имеет 2 structured и 1 skipped слот, нажимает «Совет дня»
- **THEN** совет дня выполняется, skipped-слот передаётся как «пропущен», LLM учитывает это в анализе

#### Scenario: Совет дня с пустыми слотами
- **WHEN** пользователь нажимает «Совет дня», но есть слоты со статусом `empty` (не skipped)
- **THEN** кнопка неактивна, отображается подсказка «Заполните все приёмы пищи»

### Requirement: Отображение прогресса LLM-запроса
Система SHALL показывать прогресс-бар и данные, которые будут отправлены в LLM, перед выполнением запроса.

#### Scenario: Прогресс при структуризации
- **WHEN** запущена батч-структуризация нескольких жёлтых событий
- **THEN** отображается прогресс-бар с текстом «Обработка 2 из 3 приёмов...» и список отправляемых текстов

#### Scenario: Прогресс при совете дня
- **WHEN** пользователь нажимает «Совет дня»
- **THEN** отображается прогресс-бар с текстом «Анализирую ваш день...», сводка отправляемых данных (калории, БЖУ, список блюд)

### Requirement: Батч-обработка жёлтых событий
Система SHALL иметь возможность обработать несколько событий со статусом `raw` за один LLM-запрос.

#### Scenario: Кнопка «Обработать всё»
- **WHEN** пользователь нажимает «Обработать всё» на дашборде, и есть 3 события со статусом `raw`
- **THEN** все три текста отправляются в LLM в одном запросе, ингредиенты распределяются по соответствующим `MealEvent`

### Requirement: Фоновое завершение LLM-запроса
Система SHALL завершать активный LLM-запрос при сворачивании приложения, используя `UIApplication.beginBackgroundTask`.

#### Scenario: Сворачивание во время запроса
- **WHEN** пользователь сворачивает приложение во время активного LLM-запроса
- **THEN** система регистрирует фоновую задачу (~30 секунд), запрос завершается, результат сохраняется
