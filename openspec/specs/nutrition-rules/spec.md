## Purpose

Оценка качества питания пользователя относительно 16 правил ВОЗ/ФАО (2024) через JSON-конфигурацию и единый вычислительный примитив `range`.

## Requirements

### Requirement: Каталог правил питания из JSON
Система SHALL загружать 16 правил питания ВОЗ/ФАО из `nutrition_rules.json` в bundle при запуске и предоставлять их в UI, сгруппированными по 4 категориям (макронутриенты, ограничения, качество продуктов, режим питания).

#### Scenario: Успешная загрузка правил
- **WHEN** приложение запускается и JSON корректен
- **THEN** RuleEngine загружает все правила и они отображаются в списке правил

#### Scenario: Некорректное правило в JSON
- **WHEN** правило имеет неизвестный `field`, неверный `type` или невалидный `window`
- **THEN** правило логируется и пропускается, остальные правила загружаются нормально

#### Scenario: Отсутствие JSON-файла
- **WHEN** `nutrition_rules.json` отсутствует в bundle
- **THEN** система показывает ошибку конфигурации и не показывает список правил

### Requirement: Единый примитив range с опциональными границами
Система SHALL использовать единый вычислительный примитив `range` с опциональными `lower`/`upper` и параметром `window: "day" | "week"` для оценки 14 из 16 правил.

#### Scenario: Правило с обеими границами
- **WHEN** правило имеет `lower` и `upper`, а значение в пределах границ
- **THEN** система возвращает зону `normal`

#### Scenario: Правило только с верхней границей
- **WHEN** правило имеет только `upper`, а значение превышает порог
- **THEN** система возвращает зону `violation`

#### Scenario: Правило только с нижней границей
- **WHEN** правило имеет только `lower`, а значение в warning-зоне (≥85% от порога)
- **THEN** система возвращает зону `warning`

#### Scenario: Недельное окно
- **WHEN** правило имеет `window: "week"`, а сумма за неделю превышает недельный порог
- **THEN** система возвращает зону `violation` с magnitude = отклонение от порога

### Requirement: Особые правила presence и countSkipped
Система SHALL реализовать правила `presence` (наличие категории ингредиента) и `countSkipped` (пропуски приёмов) вне range-примитива как особые случаи в RuleEngine.

#### Scenario: Presence-правило выполнено
- **WHEN** хотя бы один ингредиент за день имеет целевую `foodCategory`
- **THEN** правило показывает зону `normal`

#### Scenario: Presence-правило нарушено
- **WHEN** ни один ингредиент за день не имеет целевую `foodCategory`
- **THEN** правило показывает зону `violation`

#### Scenario: countSkipped — предупреждение
- **WHEN** количество пропущенных окон за неделю 1–3
- **THEN** правило показывает зону `warning`

#### Scenario: countSkipped — нарушение
- **WHEN** количество пропущенных окон за неделю 4 и более
- **THEN** правило показывает зону `violation`

### Requirement: Включение и выключение правил пользователем
Система SHALL хранить состояние enabled/disabled каждого правила в `NutritionRuleConfig` (SwiftData) и фильтровать выключенные правила при оценке.

#### Scenario: Выключение правила
- **WHEN** пользователь переключает toggle правила в `Выкл`
- **THEN** правило исключается из оценки и сводной статистики

#### Scenario: Включение правила после выключения
- **WHEN** пользователь переключает toggle правила в `Вкл`
- **THEN** правило снова участвует в оценке, с данными начиная с текущей недели

#### Scenario: Первый запуск — все правила включены
- **WHEN** приложение запускается впервые
- **THEN** для каждого правила из JSON создаётся `NutritionRuleConfig` с `isEnabled = true`

### Requirement: Три зоны оценки
Система SHALL классифицировать результат каждого правила в одну из трёх зон: `normal` (зелёный), `warning` (жёлтый, ≥85% порога), `violation` (красный, превышение порога).

#### Scenario: Значение в норме
- **WHEN** значение находится в допустимых границах и не достигает warning-зоны
- **THEN** правило показывает зону `normal`

#### Scenario: Warning-зона для range с двумя границами
- **WHEN** значение находится между порогом и 85% порога с любой стороны
- **THEN** правило показывает зону `warning`

#### Scenario: Warning-зона для range с одной границей
- **WHEN** значение находится между порогом и 85% порога со стороны ограничения
- **THEN** правило показывает зону `warning`

### Requirement: Оценка при любом количестве данных
Система SHALL оценивать правила питания при наличии ≥1 structured-приёма в пределах окна правила (день или неделя). Если данных нет ни за один день окна — правило показывает зону `noData`.

#### Scenario: Один приём за день
- **WHEN** сегодня есть ровно 1 structured-приём, а правило имеет `window: "day"`
- **THEN** правило оценивается по этому приёму и показывает зону согласно значению метрики

#### Scenario: Один день в неделе
- **WHEN** в текущей неделе есть ровно 1 день со structured-приёмом, а правило имеет `window: "week"`
- **THEN** правило оценивается по этому дню и показывает зону согласно агрегированному значению

#### Scenario: Нет данных за окно
- **WHEN** за день нет ни одного structured-приёма (для дневного окна) или за неделю нет ни одного structured-приёма (для недельного окна)
- **THEN** правило показывает зону `noData`

### Requirement: Контрибьюшн-анализ на уровне продукта
Система SHALL при нарушении правила (`warning` или `violation`) вычислять вклад каждого продукта в нутриент правила за день и возвращать карту вкладов (`productId → contributionPercent`). Продукты с вкладом >30% помечаются как значимые.

**Для абсолютных метрик** (sodiumMg, fiberGrams, fruitVegGrams, redMeatGrams): `contribution = itemValue / dailyTotal × 100%`.
**Для процентных метрик** (fatPercent, proteinPercent, carbsPercent, saturatedFatPercent, sugarPercent): `contribution = itemNutrientGrams / totalNutrientGrams × 100%` — доля в граммах нутриента, а не в калориях.
**Для energyBalancePercent**: `contribution = itemCalories / totalCalories × 100%`.
**Для presence-правил**: продукт целевой категории → 100%.

#### Scenario: Контрибьюшн для процентного правила — доля в нутриенте
- **WHEN** правило `added_sugar` нарушено, дневной total сахара = 14.0g, ингредиент «Пирожное» содержит 6.4g сахара
- **THEN** contribution = 6.4 / 14.0 = 45.7%, продукт помечается значимым (>30%)

#### Scenario: Контрибьюшн для абсолютного правила
- **WHEN** правило `sodium` нарушено, дневной total натрия = 3200mg, ингредиент содержит 1200mg
- **THEN** contribution = 1200 / 3200 = 37.5%, продукт помечается значимым

#### Scenario: Контрибьюшн для presence-правила
- **WHEN** правило `ultra_processed` нарушено за день и продукт имеет категорию `ultra_processed`
- **THEN** продукт получает contribution = 100% и помечается значимым

#### Scenario: Правило не нарушено
- **WHEN** правило в зоне `normal`
- **THEN** contributionMap пуста, ни один продукт не подсвечивается

#### Scenario: Вклад менее 30%
- **WHEN** правило нарушено, но вклад продукта в нутриент составляет 15%
- **THEN** продукт не помечается значимым и не подсвечивается

### Requirement: Топ-3 продукта по вкладу
Система SHALL ограничивать подсветку продуктов тремя наибольшими по вкладу в каждое нарушенное правило.

#### Scenario: Более трёх продуктов с вкладом >30%
- **WHEN** 5 продуктов имеют вклад >30% в нарушенное правило `sodium`
- **THEN** подсвечиваются только топ-3 по вкладу

#### Scenario: Менее трёх продуктов с вкладом >30%
- **WHEN** только 1 продукт имеет вклад >30%
- **THEN** подсвечивается только этот продукт

### Requirement: Вычисление процентов от калорийности
Система SHALL вычислять процентные правила (жиры, белки, сахар, углеводы) от общей калорийности дня по стандартным коэффициентам: жиры × 9, белки и углеводы × 4.

#### Scenario: Расчёт процента жиров
- **WHEN** totalCalories > 0
- **THEN** fatPercent = (fatGrams × 9) / totalCalories

#### Scenario: Нулевая калорийность
- **WHEN** totalCalories = 0 (все приёмы skipped или нет данных)
- **THEN** правило получает «нет данных» за этот день

### Requirement: Сводная статистика правил
Система SHALL показывать сводную строку «N из M правил соблюдаются» только в `DayAnalyticsPanel` на дашборде. На экране настройки правил (`RulesListView`) сводная статистика SHALL NOT отображаться. N — количество правил в зоне `normal`, M — общее количество включённых правил.

#### Scenario: Все правила в норме
- **WHEN** все 16 правил включены и показывают `normal`
- **THEN** `DayAnalyticsPanel` показывает «16 из 16 правил соблюдаются» зелёным цветом

#### Scenario: Часть правил нарушена
- **WHEN** 12 правил `normal`, 3 `warning`, 1 `violation`
- **THEN** `DayAnalyticsPanel` показывает «12 из 16 правил соблюдаются» красным цветом (худшая зона — violation)

#### Scenario: Экран правил не показывает сводную статистику
- **WHEN** пользователь открывает `RulesListView`
- **THEN** сводная строка и круговой индикатор отсутствуют, экран содержит только список правил по категориям с toggle вкл/выкл

### Requirement: Фиксированная неделя (пн–вс)
Система SHALL использовать фиксированную календарную неделю (понедельник–воскресенье) для всех правил.

#### Scenario: Сброс недельных правил в понедельник
- **WHEN** наступает понедельник
- **THEN** недельные правила (`window: "week"`) обнуляются, старые `RuleViolation` удаляются, графики начинаются заново

#### Scenario: Частичная неделя
- **WHEN** текущая неделя ещё не завершена и в ней ≥3 дней данных
- **THEN** правила оцениваются по доступным дням без ожидания воскресенья

### Requirement: Energy balance через цель пользователя
Система SHALL использовать целевую калорийность, заданную пользователем в настройках, для правила энергобаланса.

#### Scenario: Превышение цели
- **WHEN** фактическое потребление за день превышает цель пользователя
- **THEN** правило energy_balance показывает зону согласно величине отклонения и warningRatio

#### Scenario: Цель не задана
- **WHEN** пользователь не задал целевую калорийность
- **THEN** правило energy_balance показывает «Нет данных»
